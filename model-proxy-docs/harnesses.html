Agents & harnesses

Concrete env recipes for pointing real agents at the proxy — SDKs, Claude Code, Codex, and the identity conventions that keep a whole fleet's records attributable.

There is exactly one integration lever, and it's the right one: the base-URL environment variable every serious SDK already honors. No harness plugin can inject headers or env into a harness's model calls from the inside — so the recipe is always the same: set the variable in the environment that launches the agent, and identity rides the URL from there. 

   http://proxy:8080/{owner}/{agent}/{session}/{provider}/…
                      who pays   which harness  which instance  which upstream

#1. The two SDK families

OpenAI SDKs (Python, JS, and the many tools built on them) — base URL ends in /v1:

sh
export OPENAI_BASE_URL="http://proxy:8080/m64/my-agent/dev/openai/v1"
export OPENAI_API_KEY="sk-…"        # forwarded untouched (or an sk-syn virtual key)

Anthropic SDKs — the SDK appends /v1/messages itself, so the base URL ends at the provider token:

sh
export ANTHROPIC_BASE_URL="http://proxy:8080/m64/my-agent/dev/anthropic"
export ANTHROPIC_API_KEY="sk-ant-…"

Swap the {provider} token to aim the same SDK elsewhere — …/ollama/v1 for local models, …/openrouter/v1 for the multi-vendor router (providers).

#2. Claude Code

sh
export ANTHROPIC_BASE_URL="http://proxy:8080/${SYNADIA_OWNER:-$USER}/claude-code/${SYNADIA_NAME:-main}/anthropic"
claude

Claude Code honors ANTHROPIC_BASE_URL from its launch environment — every model call it makes (including sub-agents) then carries claude-code attribution in the path. Set the variable in the shell, profile, or service unit that starts Claude Code.

#3. Codex

sh
export OPENAI_BASE_URL="http://proxy:8080/${SYNADIA_OWNER:-$USER}/codex/${SYNADIA_CODEX_SESSION:-main}/openai/v1"
codex

Same pattern, openai family. When Codex runs under a fabric channel/launcher, make sure the launcher passes its environment through to the spawned process — the environment is the contract.

#4. Anything with an HTTP client

sh
curl -N "http://proxy:8080/m64/my-agent/dev/openai/v1/chat/completions" \
  -H "Authorization: Bearer $OPENAI_API_KEY" -H 'Content-Type: application/json' \
  -d '{"model":"gpt-4o-mini","stream":true,"messages":[{"role":"user","content":"hi"}]}'

The proxy is plain HTTP with the provider's own wire format on the other side — anything that can set a base URL (LangChain, LiteLLM-style clients, custom harnesses, shell scripts) integrates identically.

#5. Composing identity: the fabric's env conventions

The Synadia agents standardize identity env vars, so base URLs compose instead of being hand-written per machine. Resolution order (first non-empty wins): 

   CLI flag  >  SYNADIA_<AGENT>_OWNER  >  SYNADIA_OWNER  >  legacy vars  >  config  >  $USER
                └ per-agent override      └ fleet-wide ("set once for everything")
sh
# one fleet-wide identity in your profile:
export SYNADIA_OWNER="m64"
# then per-harness base URLs compose from it:
export OPENAI_BASE_URL="http://proxy:8080/${SYNADIA_OWNER}/codex/${SYNADIA_CODEX_SESSION:-main}/openai/v1"
export ANTHROPIC_BASE_URL="http://proxy:8080/${SYNADIA_OWNER}/claude-code/${SYNADIA_CLAUDE_CODE_NAME:-main}/anthropic"
  • Keep the agent token = the harness's canonical name (claude-code, codex, opencode, …). It identifies the harness, not the deployment — the same convention the fabric's NATS channels use.
  • session names the instance — dev, ci, a project name. Records and subjects stay per-instance attributable; collisions are fine (it's attribution, not a lock).
  • Tokens must match [a-zA-Z0-9_-]{1,63} — they become NATS subject tokens.

#6. When the URL isn't enough: header overrides

Some clients can set headers but not the URL path (or you want one base URL behind a shared gateway). The proxy consumes these — they never reach the provider:

headereffect
Synadia-Owner / Synadia-Agent / Synadia-Sessionoverride the corresponding path token
Synadia-Correlation-Idechoed into the record's corr_id — join model calls with the fabric's other lenses (wire audit, durable runs)

#7. Two dials worth setting fleet-wide

  • Opt into usage yourself where you can. Harnesses that let you set request options should send stream_options: {"include_usage": true} once and keep it — those streams stay 100% byte-faithful (stream_opt_in), and the proxy's injection remains the safety net for everything else (why).
  • Prefer virtual keys for fleet deployments. export OPENAI_API_KEY='sk-syn-v1-…' and the agent never holds a real secret — per-agent attribution, millisecond revocation, provider keys sealed server-side (virtual keys).

The bypass guarantee, one more time: every recipe on this page is one unset away from direct traffic. The proxy holds no state your agent depends on — adopting it is reversible by construction, per agent, per shell.
