How state is persisted

There is no database and no durable-execution server. A workflow's entire state is an ordered, append-only log on NATS JetStream — and one append primitive is the whole correctness story.

#The substrate: four NATS resources

All durability lives in NATS. The library provisions four resources (idempotently) and never needs anything else — no Postgres, no Redis, no control plane. Your only processes are NATS and your own workers with the library embedded.

ResourceKindHoldsRole
DE_JOURNALstream (file)de.journal.{run} — one ordered subject per runThe source of truth. Every step, value, signal, and terminal result.
DE_WORKstream (work queue)de.work.{wf} — start / resume / signal / child-done items“Make progress on run X.” One durable pull consumer per workflow, load-balanced across workers.
DE_TIMERSstream (scheduled)de.timer.> — a native JetStream schedule + its wake targetFires ctx.sleep / ctx.await timeouts as a future wake.
DE_RUNSKV bucket{run} → { status, wf, version, result? }A fast status index / hint — never trusted for a progress decision.

#State is a log, not rows

Most engines store workflow state as mutable rows you read and update. Here, state is the journal: an append-only list of entries on the run's own subject, in the exact order they happened. You never update an entry — you only append the next one. To know a run's state, you read its log from the start; to make progress, you append.

Because the log is ordered and immutable, reconstructing state is deterministic and a crash is harmless: whatever was appended is durable, and whatever wasn't simply didn't happen. The wire layout of these entries is frozen at v:1 and specified language-neutrally in the wire protocol, so a Go or Python worker could read the very same log.

#The one primitive: optimistic append

Everything that must be correct under concurrency reduces to a single NATS capability — an optimistic (compare-and-swap) append. When publishing to a run's journal subject, you assert the sequence you expect to be appending after: 

ts
// "accept this append only if the subject's last entry is still sequence S"
await js.publish(subject, data, {
  expect: { lastSubjectSequence: S },
  msgID: "…", // dedup: a redelivered identical append is recognized, not duplicated
});

If another worker already appended, the server rejects yours with one specific error — 10071, “wrong last sequence” — which the library maps to a FencedError and handles everywhere. Progress is never gated on a plain read (a read can be served by a lagging replica); it is always an ordered append or a CAS. KV create / update(rev) is the same primitive under the hood, since a KV bucket is itself a stream.

One mechanism, one error code. The journal append, the KV index write, the idempotent start, the timer schedule — all the same optimistic-append/CAS, all surfacing the same 10071 when they lose a race.

#Two-phase steps: persist intent before the effect

A durable step (ctx.run) is written in two phases. The step's intent (step-begin) is CAS-appended before the side effect runs; the effect runs; then the result (step-end) is appended. This ordering is what makes a redelivered or duplicated worker safe:

Because the log elects the writer (not the queue), each step is journaled exactly once — the workflow's result is deterministic no matter how many workers briefly touched the run. What is at-least-once is the step function's execution: across the begin → end window a crash, an ack-timeout redelivery, or a second worker on a concurrent wake can re-run fn. That is the standard durable-execution contract — exactly-once state, at-least-once effects — and the reason a non-idempotent effect (charging a card, sending an email) should key on ctx.key() (${runId}:${seq}, stable across re-runs): 

ts
await ctx.run("charge", () =>
  stripe.charges.create(
    { amount, currency: "usd", source },
    { idempotencyKey: ctx.key() }, // same key on every re-run of this step
  ),
);

#The life of a write

Putting it together: starting a run and advancing it are both just CAS appends, with the work queue deciding only who gets to try next — never what the state is.

  1. Start appends a start entry (idempotent on runId) and drops a work item on DE_WORK.
  2. A free worker pulls the item and loads the journal.
  3. It replays the function from the top: journaled ctx.* calls return their saved value; the first un-journaled one is the live frontier — it runs for real and is appended.
  4. At a terminal (result / failed / continued) the worker appends it, updates the DE_RUNS hint, and acks. At a sleep / await it suspends; a later wake re-enqueues the run.
  5. If the worker dies anywhere, the item is redelivered once its ack lease expires and another worker resumes from step 2 — the journal is unchanged, so nothing is lost or repeated in state.

#Why the KV index is only a hint

DE_RUNS exists so handle.status() and the dashboard can answer “what's the state of this run?” cheaply, without scanning a journal. It is written after the journal as a best-effort convenience. Crucially, the runtime never reads it to make a decision — if a worker dies after appending a terminal but before updating the KV, the status hint is stale yet the run is genuinely done; handle.result() reads the journal (the truth) and resolves correctly. The index can lag or be wrong; the log cannot.

#Timers and signals are persisted too

Waiting is durable. ctx.sleep(name, ms) uses native JetStream message scheduling: the wake is a real message the server delivers at the deadline — it survives worker restarts, because nothing in your process is holding the timer. ctx.await(name) parks the run until client.signal(runId, name, data) appends a signal entry; with timeoutMs a durable timer races the signal, and signal-vs-timeout is resolved deterministically by journal order. A parked run consumes no worker — it is just a log waiting for its next entry.

#Durability, bounds & retention

  • File storage, replicated. Streams are file-backed; on a NATS cluster the journal inherits JetStream replication, so durability is whatever your NATS already guarantees.
  • Bounded per run. Each stream sets max_bytes (required on NGS). For long-running or looping workflows, ctx.continueAsNew(input) ends the current generation and starts a fresh one with an empty journal — keeping any single run's log bounded no matter how long it runs.
  • 8 MiB payloads. An entry (and any step result) is bounded by the NATS max payload; object-store offload for larger values is a documented non-goal for now.
  • No external state. Workers are stateless; restart them freely. Everything needed to resume a run is in NATS.

Want the exact entry kinds, append rules, and algorithms? See the wire protocol and the deep-dive design (§§3–7), or read a journal as it is built in the journal-on-the-wire walkthrough.
