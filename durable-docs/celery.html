Coming from Celery

Celery is a great task queue for firing off independent background jobs. Where it stops is durable multi-step work — a pipeline that must survive a crash mid-sequence without losing a step or repeating one. That's exactly what a durable workflow gives you.

A note on language. Celery is Python; @synadia-ai/durable is a TypeScript library today. The on-the-wire format is language-neutral (a Python SDK could ride the same NATS), but it doesn't exist yet — so “replace” here means running that workflow on a small TypeScript worker alongside your Python services, all on the NATS you already operate.

#A task → a durable step

A Celery task becomes a ctx.run step inside a workflow. You enqueue with client.start and run a worker — the broker and result backend are both just your NATS.

Celery · Python
python
# tasks.py
from celery import Celery

app = Celery("billing", broker="redis://localhost")

@app.task
def charge_card(cart_id, amount):
    return stripe.Charge.create(amount=amount, source=cart_id)

# enqueue from your app
charge_card.delay("c_1", 42)

# run a worker:  $ celery -A tasks worker
@synadia-ai/durable
durable
import { connect } from "@nats-io/transport-node";
import { createClient, createWorker, defineWorkflow } from "@synadia-ai/durable";

const charge = defineWorkflow("charge", async (ctx, input: { cartId: string; amount: number }) =>
  ctx.run("charge-card", () =>
    stripe.charges.create({ amount: input.amount, source: input.cartId }),
  ),
);

const nc = await connect({ servers: "nats://127.0.0.1:4222" });

// enqueue from your app
const client = createClient({ nc });
await client.start(charge, { cartId: "c_1", amount: 42 });

// run a worker
await createWorker({ nc, workflows: [charge] }).start();

#Retries → built-in, with real idempotency

Celery retries re-run the whole task — and since delivery is at-least-once, a non-idempotent task can fire twice. ctx.run takes a retry policy too, but the durable model also hands you a stable idempotency key (ctx.key()) so the effect is safe across retries and redeliveries.

Celery · Python
python
@app.task(bind=True, max_retries=5, default_retry_delay=2)
def charge(self, order_id, amount):
    try:
        # at-least-once: this can run more than once
        return gateway.charge(order_id, amount)
    except TransientError as exc:
        raise self.retry(exc=exc)
@synadia-ai/durable
durable
await ctx.run(
  "charge",
  () => gateway.charge(orderId, amount, { idempotencyKey: ctx.key() }),
  { retry: { kind: "exponential", attempts: 5, baseMs: 2000, factor: 2 } },
);

#Delayed / scheduled → ctx.sleep

A durable timer: the wake is a scheduled NATS message, so a parked run survives worker restarts and consumes no worker while it waits.

Celery · Python
python
# run in one hour
send_reminder.apply_async((user_id,), countdown=3600)
@synadia-ai/durable
durable
await ctx.sleep("cool-off", 60 * 60 * 1000);
await ctx.run("reminder", () => sendReminder(userId));

#Chains → one crash-safe workflow

This is the headline. A Celery chain wires tasks into a pipeline, but if the worker dies after authorize there's no built-in resume — the rest is lost, and a naive requeue re-runs earlier tasks (double-auth). A durable workflow is just an async function; on a crash a fresh worker replays the journal and continues at the first unfinished step.

Celery · Python
python
from celery import chain

# reserve → authorize → capture
chain(reserve.s(cart), authorize.s(amount), capture.s()).apply_async()

# crash after `authorize`?  `capture` never runs, and a requeue
# re-executes `reserve`/`authorize` — you hand-roll state + dedup.
@synadia-ai/durable
durable
const checkout = defineWorkflow("checkout", async (ctx, input: Cart) => {
  const orderId = await ctx.run("reserve", () => reserve(input.cart));
  const authId = await ctx.run("authorize", () => authorize(orderId, input.amount));
  return ctx.run("capture", () => capture(authId));
});

// crash after `authorize`? a fresh worker replays reserve + authorize from
// the journal (no re-run) and resumes at `capture`. Exactly-once state.

#Periodic (beat) → continue-as-new

Instead of an external scheduler, a long-running workflow does its work, sleeps a period, and continueAsNews into the next cycle with a fresh, bounded journal — running “forever” as one logical run.

Celery · Python
python
# celeryconfig.py — Celery beat
from celery.schedules import crontab

app.conf.beat_schedule = {
    "nightly-rollup": {
        "task": "tasks.rollup",
        "schedule": crontab(hour=0, minute=0),
    },
}
@synadia-ai/durable
durable
const rollup = defineWorkflow("rollup", async (ctx, s: { day: number }) => {
  await ctx.run("rollup", () => doRollup(s.day));
  await ctx.sleep("one-day", 24 * 60 * 60 * 1000);
  return ctx.continueAsNew({ day: s.day + 1 });
});

#When durable fits — and when Celery is still the tool

  • Reach for durable when a “job” is really a multi-step process that must not lose a step or repeat an effect across a crash — orders, payments, provisioning, sagas, human-in-the-loop approvals.
  • Stay on Celery for very high-throughput, independent, fire-and-forget tasks where its mature ecosystem, autoscaling, and Python-native story are the point. durable is a durable-execution library, not a drop-in high-volume task queue.
  • Fan-out. group / chord map to ctx.startCall per child + native Promise.all / ctx.race — durable fan-out is built in. (Per-branch local-step journaling, ctx.startRun, is the deferred bit; wrap parallel local work in one ctx.run(() => Promise.all([...])).)
  • Language. It's TypeScript today; a Python SDK over the same wire format is a future, not a shipped, thing.
