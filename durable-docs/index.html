v0.2 · durable execution on NATS / JetStream

Workflows that survive crashes — with no new server to run.

@synadia-ai/durable is a lightweight durable-execution library on NATS / JetStream — a library, not a server. Write workflows as ordinary async functions; every side-effecting or nondeterministic step is journaled to an ordered log and replayed after a crash instead of being re-executed. Your NATS cluster is the engine.

How it works See crash → replay GitHub
Why this exists

Imagine a function that reserves inventory, charges a card, and ships an order. If the process crashes after charging but before shipping, a naïve retry double-charges and a non-retry loses the order. You end up hand-rolling state machines, status columns, and idempotency tables.

Durable execution makes that function survive crashes on its own. You write normal code; on a restart the runtime picks up exactly where it left off — completed steps are not run again, and the unfinished one resumes. Most systems deliver this with a dedicated server you operate, scale, and back with a database. This library doesn't: the NATS cluster is the engine.

Traditional durable execution

A server you operate

  • A dedicated control plane to deploy, scale, patch & secure
  • A separate database (Postgres / Cassandra) behind it
  • A network hop to a control plane to start a workflow
@synadia-ai/durable

Just NATS + your workers

  • No new service — JetStream streams + KV are the journal, queue, timer scheduler & status index
  • Stateless workers; all durability lives in NATS
  • It's a library — embed it, no extra hop
  • A language-neutral wire format, so Go / Python / … SDKs can follow

“No server” doesn't mean “no infrastructure.” NATS is infrastructure. It means no new, durable-execution-specific service. If you already run NATS, you already run the engine.

Quickstart

Install the library plus the NATS clients (bring your own connection):

sh
npm install @synadia-ai/durable @nats-io/jetstream @nats-io/kv
npm install @nats-io/transport-node   # bring your own connection

Define a workflow, run a worker, start a run — and get a result that survives restarts:

ts
import { connect } from "@nats-io/transport-node";
import { createClient, createWorker, defineWorkflow } from "@synadia-ai/durable";

const charge = defineWorkflow("charge", async (ctx, input: { cartId: string; amount: number }) => {
  const orderId = await ctx.run("reserve", () => reserveInventory(input.cartId));
  const authId = await ctx.run("authorize", () => authorize(orderId, input.amount));
  return { receipt: await ctx.run("capture", () => capture(authId)) };
});

const nc = await connect({ servers: "nats://127.0.0.1:4222" });

const worker = createWorker({ nc, workflows: [charge] });
await worker.start();

const client = createClient({ nc });
const handle = await client.start(charge, { cartId: "c_1", amount: 42 });
console.log(await handle.result()); // resolves when the run completes — survives worker restarts

If the worker crashes mid-run, a restart replays reserve / authorize from the journal (without re-executing them) and resumes from the first unfinished step — see the crash → replay walkthrough. A runnable version is in examples/charge.ts.

The model in one breath
  • A workflow is a normal async function over a ctx. Anything nondeterministic or side-effecting goes through ctx.* and is journaled — ctx.run(name, fn) for steps, ctx.now / random / uuid() for determinism, ctx.sleep / await / call / continueAsNew for the rest — plus durable fan-out (ctx.startCall + Promise.all / ctx.race) and cancellation (client.cancel / ctx.onCancel).
  • On replay the function re-runs from the top: journaled ctx.* calls return their saved value (no re-execution); the first un-journaled call is the live frontier.
  • Two-phase steps. A step's intent is CAS-appended before the side effect, so a fresh zombie worker is fenced before it can fire, and every step is journaled exactly once. The whole correctness story is one NATS primitive: an optimistic append (expect.lastSubjectSequence) whose lost race is a single error code, 10071.
  • Exactly-once state, at-least-once effects — the standard durable-execution contract. ctx.key() (${runId}:${seq}) is the idempotency key that makes your effect safe.
