Wire protocol

The language-neutral, RFC-2119 wire specification (v:1) — for anyone writing a Go / Python / … SDK that interoperates on the same NATS streams.

Wire version: 1 (v:1) Library: @synadia-ai/durable (reference implementation) Status: Draft Date: 2026-06-17

#1. Introduction

This document defines a protocol for executing durable workflows over NATS JetStream. A workflow is an ordinary function whose every nondeterministic or side-effecting operation is recorded to an append-only log; on a crash the function re-runs from the top and recorded operations are replayed (returned from the log) instead of re-executed. Workers that follow this protocol journal runs, distribute work, schedule timers, route signals, and invoke sub-workflows using a uniform set of NATS resources and message shapes — so an implementation in any language interoperates with another over the same streams.

There is no durable-execution server. The NATS cluster is the engine: JetStream streams are the journal, the work queue, and the timer scheduler; a KV bucket is the status index. The participants are a worker (executes and journals runs) and a client (starts, observes, and signals runs).

Built on three NATS primitives, used as-is:

  • JetStream streams — ordered, append-only logs with optimistic concurrency (expect) and message deduplication (Nats-Msg-Id); a work-queue stream for at-least-once dispatch; and native message scheduling for timers.
  • JetStream KV — a revisioned key/value bucket (itself a stream) for the status index.
  • Core publish/subscribe & request/reply — addressing and routing by subject.

The protocol adds only what is missing: a resource layout (§3), a subject convention (§4), the journal wire format (§5), the work-item format (§6), the timer / signal / sub-workflow conventions (§7–§9), and the algorithms that tie them together (§11).

Out of scope for wire v1:

  • In-code patching (patched()/getVersion() — evolving an in-flight run in place). In-flight versioning itself is in scope and shipped, via version-pinned dispatch (§15).
  • Parallel fan-out (no durable Future; ctx.run is blocking).
  • Payloads larger than the NATS max_payload (8 MiB default); no object-store offload.

#1.1 Conventions

Normative keywords — MUST, MUST NOT, SHOULD, SHOULD NOT, MAY — follow RFC 2119 and RFC 8174.

"The protocol" is the wire contract defined here. "An implementation" (or "SDK") is a language-specific library built on top; the TypeScript @synadia-ai/durable package is the reference implementation. Design rationale and prose algorithms live in DESIGN.md; this document is the normative wire contract.

A run is one execution of a workflow, identified by a runId. A workflow is identified by a name. A command is a single ctx.* operation a workflow issues (it occupies a monotonic per-run sequence number, §5.2).

#1.2 Version

The wire format is frozen at v:1. Every journal entry carries "v": 1 (§5.1); a decoder MUST reject an entry whose v is not 1. The format is deliberately small and explicit so it can be an inter-language contract — the journal, not any one SDK, is the standard.

The library implementing this protocol is independently versioned (pre-1.0) and may add capabilities without changing the wire format; wire-breaking changes bump v. Compatibility rules are in §15.

#1.3 Roles

Role Responsibilities
Client Start a run (idempotently), observe it (result / status), and signal it. Holds no run state. (§11.1, §11.5–6, §9)
Worker Bind the per-workflow consumers, drive each work item by loading the journal and executing to the next halt, journal entries, schedule timers, and deliver sub-workflow results. (§6, §11.2–4)

A process MAY be a worker, a client, or both. Multiple workers MAY run concurrently for the same workflow (§6.3); correctness does not depend on their count.

#2. Foundations — the one primitive

Everything that must be correct under concurrency reduces to a single NATS operation: the optimistic append (compare-and-set, "CAS").

#2.1 Optimistic append (CAS)

A write to a journal subject MUST be a JetStream publish carrying both an expected sequence and a dedup id:

publish(subject, payload, {
  msgID: <dedup id>,                              # Nats-Msg-Id header
  expect: { lastSubjectSequence: <S> }            # Nats-Expected-Last-Subject-Sequence header
})
  • expect.lastSubjectSequence: S asserts "the last message currently on this subject has stream sequence S." If another writer has appended since, the server rejects the publish with JetStreamApiError code 10071 (StreamWrongLastSequence). This is the single-writer fence: the rejected writer has lost the race and MUST stop (it is a zombie). Implementations surface this as a FencedError.
  • msgID (the Nats-Msg-Id header) is JetStream deduplication: a redelivered identical publish lands at most once. It is belt-and-suspenders on top of the CAS — an ambiguous ack can be retried safely.

KV writes are the same primitive (a KV bucket is a stream): create / update(revision) is a CAS.

#2.2 No progress on a plain read

A worker MUST NOT gate progress on a plain read of mutable state (a kv.get, a non-CAS stream read) — in a clustered deployment such a read may be served by a replica and be stale. Every progress decision is either an ordered replay of the journal (§11.2) or a CAS append (§2.1). The status index (§10) is a read-only hint and never gates progress.

This single primitive is the whole concurrency model: no locks, no leader election, no coordinator.

#3. Resources

An implementation provisions five JetStream resources, named from a configurable prefix (default de; uppercased for stream/bucket names). Provisioning MUST be idempotent (create-if-absent).

Resource Kind Subjects / keys Required config
DE_JOURNAL Stream de.journal.* (one token = runId) File storage; Limits retention; max_bytes set.
DE_WORK Stream de.work.* (one token = workflow) File storage; WorkQueue retention; max_bytes set.
DE_TIMERS Stream de.timer.> File storage; Limits retention; max_bytes set; allow_msg_schedules: true.
DE_RUNS KV bucket key = runId File storage; history: 1; max_bytes set.
DE_LEASES KV bucket key = runId OPTIONAL (the gating run lease — advisory, never a correctness mechanism). File storage; history: 1; small max_bytes; a max-age TTL as the GC backstop.

Requirements:

  • Every stream MUST set max_bytes (NGS / R1 streams require it). On a single replica, file storage is required. max_bytes is a reservation, not on-demand — sizing is the deployer's concern.
  • DE_WORK MUST use WorkQueue retention: a message is removed once acknowledged, so the stream's message count is the live backlog.
  • DE_TIMERS MUST enable native message scheduling (allow_msg_schedules). A scheduled message's wake target MUST be a subject in the same stream (cross-stream schedule targets are rejected by the server) — hence both the schedule and its wake live under de.timer.> (§7).
  • DE_RUNS history is 1 (only the latest record per run is retained; it is a hint, §10).

A non-default prefix P replaces de/DE throughout (P.journal.*, P_JOURNAL, …). All participants sharing a logical deployment MUST use the same prefix.

#4. Subjects & identifiers

#4.1 Subjects

Subject Token(s) Purpose
de.journal.{runId} runId The run's append-only journal (the source of truth, §5).
de.work.{workflow} workflow The work queue for a workflow (start / wake items, §6).
de.timer.sched.{runId}.{seq} runId, seq A timer's schedule-definition message (§7).
de.timer.wake.{workflow} workflow Where a fired timer delivers its wake work item (§7).
KV key {runId} in DE_RUNS runId The run's status record (a hint, §10).

#4.2 Identifier rules

A runId and a workflow name are each used both as a subject token and as a KV key. KV keys forbid some characters that subjects allow (notably ~), so both MUST match the safe intersection:

^[A-Za-z0-9_-]+$        # ASCII letters, digits, '-', '_'

An implementation MUST reject any runId or workflow name outside this set before use.

#4.3 Derived runIds

Two run kinds derive their runId deterministically (both stay within §4.2):

  • continue-as-new generations (§11.4): generation 0 is the base runId itself; generation n > 0 is {base}_g{n} (e.g. sub-acme, sub-acme_g1, sub-acme_g2).
  • sub-workflow children (§9): a child started by ctx.call at command seq defaults to {parentRunId}_{seq} (e.g. checkout-ok_4). An implementation MAY allow an explicit child runId override.

#4.4 Examples & wildcards

de.journal.checkout-ok            # one run's journal
de.work.checkout                  # the "checkout" workflow's work queue
de.timer.wake.subscription        # fired timers for the "subscription" workflow
de.journal.>                      # every run's journal (admin / export)
de.work.>                         # every workflow's queue
de.timer.>                        # all timer schedules + wakes

#5. The journal (wire format v:1)

A run's journal is the ordered sequence of messages on de.journal.{runId}. It is the sole source of truth for the run; everything else (the status index, the work queue) is derived or advisory.

#5.1 Entry envelope

Every entry is a JSON object encoded as compact UTF-8 (the default codec; an implementation MAY support another codec but v:1 denotes JSON). Every entry MUST carry:

Field Type Description
v number Wire version. MUST be 1. Decoders MUST reject any other value.
kind string Entry discriminator (§5.3). Decoders MUST reject an unknown kind.
seq number Command position (§5.2). -1 for non-command entries.
ts number Wall-clock ms at write time. Informational only — implementations MUST NOT use ts for any control decision.

Additional fields are per-kind (§5.3). Decoders MUST reject an entry missing v, kind, or seq.

#5.2 Command sequence

seq is the command position: a per-run counter, 0, 1, 2, …, assigned in workflow-execution order to each ctx.* command (run, now, random, uuid, sleep, await, call). Replay matches a command to its recorded entry by this position (§11.2).

  • A two-phase step (ctx.run) shares one seq across its step-begin and step-end entries.
  • Event entries that record something external — signal, resolve — and terminal entries — result, fail, continued — do not consume a command position; they carry seq = -1 (NON_COMMAND_SEQ). A resolve references the command it completes via resolvesSeq.

#5.3 Entry kinds

The sixteen kinds and their per-kind fields:

start — the first entry of every run (seq 0):

Field Type Req Description
wf string Yes Workflow name.
wfVersion number Yes Workflow version recorded at start.
input any Yes The run's input (JSON-serializable).
parent object No Sub-workflow link: { runId, seq, wf } of the caller (§9).
generation number No continue-as-new generation; default 0 (§11.4).
base string No The chain's generation-0 runId; default = this runId (§11.4).

step-begin — a step's intent, appended before the side effect runs:

Field Type Req Description
name string Yes Step name (matched on replay).
key string Yes Idempotency key for the effect: {runId}:{seq} (§14).

step-end — a step's recorded result (same seq as its step-begin):

Field Type Req Description
name string Yes Step name (MUST match the begin).
value any Yes The step's return value.

now / random / uuid — a captured deterministic value:

Field Type Req Description
value any Yes The captured value (ms number / float / uuid string).

sleep — a durable timer command:

Field Type Req Description
name string Yes Timer name.
dueMs number Yes Absolute deadline (ms since epoch); deterministic once recorded.
schedSubj string No Subject of the native schedule that fires the wake (§7).

await — block until a signal of name arrives:

Field Type Req Description
name string Yes Signal name to wait for.
dueMs number No Absolute timeout deadline, if ctx.await was given a timeoutMs (§8).

signal (non-command, seq -1) — a delivered external signal:

Field Type Req Description
name string Yes Signal name (matched FIFO to a same-named await).
data any Yes Signal payload.
signalId string Yes Dedup id so a redelivered signal is appended once.

call — invoke a sub-workflow:

Field Type Req Description
name string Yes Sub-workflow name.
childRunId string Yes The child run's id (§4.3).
wf string Yes Sub-workflow name (same as name).
input any Yes The child's input.

resolve (non-command, seq -1) — completes a sleep / await / call:

Field Type Req Description
resolvesSeq number Yes The command seq being completed.
value any No Result value (for a completed call).
error object No { name, message } for a failed sub-workflow (§9).

A resolve with neither value nor error completes a sleep (timer fired) or a timed-out await (§8). A resolve carrying value/error completes a call.

result (terminal, seq -1) — the run returned:

Field Type Req Description
value any Yes The run's return value.

fail (terminal, seq -1) — the run failed:

Field Type Req Description
error object Yes { name, message } of the logic error (§12).

continued (terminal, seq -1) — continue-as-new started a fresh generation:

Field Type Req Description
nextRunId string Yes The next generation's runId ({base}_g{generation}).
input any Yes Input handed to the next generation.
generation number Yes The next generation's number (this gen + 1).
base string Yes The chain's generation-0 runId.

cancel-requested (non-command, seq -1) — client.cancel was delivered; appended before replay so the next suspend frontier observes it (§11.7):

Field Type Req Description
reason string No Optional human-readable reason from the caller.

cancelled (terminal, seq -1) — the run was hard-stopped at a suspend frontier (§11.7):

Field Type Req Description
reason string No Optional reason carried from the cancel-requested.

#5.4 Append rules — msgID & expect

Each entry is appended with the §2.1 CAS. The expect.lastSubjectSequence is the stream sequence of the previous entry on this subject (0 for the first, the start). The msgID (Nats-Msg-Id) follows a fixed convention so a retried publish is idempotent:

Entry msgID
start {runId}:start
step-begin {runId}:{seq}:b
step-end {runId}:{seq}:e
now / random / uuid {runId}:{seq}
sleep / await / call {runId}:{seq}
signal {runId}:sig:{signalId}
resolve {runId}:{seq}:r
result / fail / continued / cancelled {runId}:terminal

The single {runId}:terminal id for all terminal kinds ensures a run records at most one terminal entry: a second writer attempting a different terminal is deduplicated, and the journal's terminal is canonical.

#5.5 Reading a journal

To load a run, read all messages on de.journal.{runId} in stream order and decode each (§5.1). The ordered list is the run's full state. A run is terminal iff it contains a result, fail, or continued entry; it has at most one such entry (§5.4).

#6. The work queue

Work is dispatched as work items: JSON messages on de.work.{workflow}, consumed by that workflow's durable consumer.

#6.1 Work-item shape

json
{ "runId": "checkout-ok", "kind": "start" }
Field Type Req Description
runId string Yes The run this item drives.
kind string Yes One of start, timer, signal, child-done, cancel (and the reserved resume, §6.5).
resolvesSeq number For timer / child-done The command seq this wake completes.
signal object For signal { name, data, id } — the inbound signal (id gives idempotency).
child object For child-done The finished sub-workflow's outcome: { id, ok: true, value } or { id, ok: false, error } (id = childRunId).

#6.2 Work-item msgIDs

Item Subject msgID Notes
start de.work.{workflow} {runId}:work:start Enqueued once per new run / generation.
signal de.work.{workflow} {runId}:sig:{id} id is the signal's uuid.
child-done de.work.{parentWf} {childRunId}:done Delivered to the parent's queue (§9).
timer (delivered by schedule) schedule id {runId}:sched:{seq} Fired by native scheduling (§7).

#6.3 Consumers

For each registered workflow, a worker binds one durable pull consumer on DE_WORK:

Property Value
Durable name de-worker-{workflow}
Filter subject de.work.{workflow}
Ack policy Explicit
Ack wait Implementation-configurable; default 30 s.
Max deliver -1 (unlimited redelivery)

Multiple worker processes binding the same durable name share the consumer; JetStream load-balances items across them (this is horizontal scale — add a worker, get throughput). A run is processed by exactly one worker at a time per item; correctness across overlap rests on the journal CAS (§2), not on the consumer count.

#6.4 Acknowledgement semantics

After driving a work item (§11.2), the worker MUST:

Outcome Action Why
Run reached a terminal (result/fail) ack Done; (notify parent first if a sub-workflow, §9).
Run did continueAsNew ack Generation done; next generation already enqueued (§11.4).
Run suspended (sleep/await/call) ack A wake (timer/signal/child-done) will re-enqueue it.
Fenced (lost a CAS race) term Another worker owns this run; abandon this delivery.
Unparseable item / no journal for runId term Nothing to drive.
Infrastructure error (non-fenced) nak Transient; redeliver and replay (safe).

A suspended run is acked, not held: it occupies no worker and no memory until a wake re-enqueues it.

#6.5 The resume kind

resume is reserved in the work-item schema for an explicit re-drive. Wire v1 workers do not emit it (wakes use timer / signal / child-done); implementations MUST accept the schema field and MAY treat a resume item as a no-payload re-drive of the run.

#7. Timers (ctx.sleep and await timeouts)

A durable timer rides native JetStream message scheduling. To arm a wake at absolute dueMs for run runId's command seq:

  1. Construct the wake work item: { runId, kind: "timer", resolvesSeq: seq }.
  2. Publish it to de.timer.sched.{runId}.{seq} with a single-shot schedule firing at dueMs and targeting de.timer.wake.{workflow}, deduplicated by msgID = {runId}:sched:{seq}:
publish("de.timer.sched.{runId}.{seq}", <timer work item>, {
  schedule: { specification: <Date(dueMs)>, target: "de.timer.wake.{workflow}" },
  msgID:    "{runId}:sched:{seq}"
})

The msgID makes re-arming the same (runId, seq) on replay idempotent — a step that replays does not create a second wake. Because schedule targets MUST be in the same stream, the wake subject lives under de.timer.> (§3).

When the schedule fires, the server delivers the timer work item to de.timer.wake.{workflow}. The worker binds a second durable pull consumer per workflow for these:

Property Value
Durable name de-timer-{workflow}
Filter subject de.timer.wake.{workflow}
Ack policy Explicit; same ack-wait as §6.3

On receiving a timer item the worker appends a resolve (resolvesSeq = seq, no value) to the journal if not already present (idempotent via msgID {runId}:{seq}:r), then re-drives the run (§11.3). The deadline is journaled (the sleep/await entry's dueMs), so it survives restarts regardless of which worker is alive when it fires.

Native scheduling requires a NATS server with message scheduling support (the reference deployment pins nats-server ≥ 2.14). The deadline lives in the journal and the schedule; no process holds it in memory.

#8. Signals (ctx.await + client.signal)

A signal is external input delivered to a parked run.

  1. Send. client.signal(runId, name, data) resolves the run's workflow (via the status index, §10, or the run's start entry) and publishes a signal work item to de.work.{workflow}: { runId, kind: "signal", signal: { name, data, id } } with msgID = {runId}:sig:{id} (a fresh uuid id).
  2. Record. On receiving it, the worker appends a signal journal entry (§5.3) before replay, if no entry with that signalId exists (idempotent via msgID {runId}:sig:{signalId}), then re-drives.
  3. Match. During replay, ctx.await(name) matches the earliest unconsumed signal of that name (FIFO per name), in journal order. The match is deterministic, so replay always pairs the same signal with the same await.

Timeout. ctx.await(name, { timeoutMs }) records dueMs on the await entry and also arms a timer (§7). The signal and the timer race; whichever produces a journal entry first wins, deterministically by journal order. A timeout (a resolve with no matching signal) makes ctx.await throw a catchable DurableTimeoutError (§12) — control flow, not a failure.

#9. Sub-workflows (ctx.call)

ctx.call(workflow, input) runs a child workflow and resumes the parent with its result.

  1. Record. The parent appends a call entry (childRunId, wf, input) at its command seq.
  2. Start child. The worker starts the child run idempotently (§11.1) — appending the child's start entry with a parent link { runId, seq, wf } — and enqueues the child's start work item. The parent then suspends (§6.4).
  3. Deliver result. When the child reaches a terminal, the worker reads the child's parent link and publishes a child-done work item to the parent's queue de.work.{parentWf} with msgID = {childRunId}:done: { runId: parent.runId, kind: "child-done", resolvesSeq: parent.seq, child: { id: childRunId, ok, value | error } }.
  4. Resume parent. The worker appends a resolve (with value on success, or error on failure) for the parent's call seq, then re-drives. On success ctx.call returns the value; on failure it throws a WorkflowFailure (a child failure fails the parent unless caught — §12, and the basis for saga compensation).

A child is an ordinary run with its own journal and consumer; sub-workflows compose to any depth. The parent link threads forward across a child's own continueAsNew (§11.4).

#10. Status index (DE_RUNS KV)

The worker maintains a per-run record in the DE_RUNS KV (key = runId) as a fast hint for observers. It is never read to gate progress (§2.2); the journal is the truth.

json
{ "status": "blocked", "wf": "purchase", "version": 2, "detail": "waiting for purchase v2", "updatedMs": 1781667505735 }
Field Type Description
status string One of pending, running, suspended, blocked, completed, failed, continued, cancelled. blocked = the run's recorded wfVersion isn't registered on any worker yet (§15); it parks and resumes when that version is deployed.
wf string Workflow name.
version number Workflow version.
updatedMs number Last update (ms since epoch).
result any Present when completed.
error object { name, message }, present when failed.
next string Next generation runId, present when continued.

Writes are plain puts. A suspended write MUST NOT downgrade a record already in a terminal status (completed/failed/continued/cancelled) — the journal terminal wins. A stale read can race a concurrent finalize (the gating lease is advisory and gates only execution), but a terminal status is never clobbered back to suspended, and observers that need the truth read the journal.

#11. Algorithms

#11.1 Start (idempotent)

  1. Validate runId and workflow name (§4.2).
  2. Provision resources (§3) if not already.
  3. Append the start entry with expect.lastSubjectSequence = 0 and msgID = {runId}:start (§5.4).
  4. If the append was not a duplicate: write status: running (§10) and enqueue a start work item (§6.2). If it was a duplicate, or the append was fenced (10071): the run already exists — do nothing further.
  5. Return a handle bound to runId.

Starting the same runId twice is therefore a no-op: the runId is the idempotency key for start.

#11.2 Execute / replay (two-phase, intent-first)

A worker drives a work item by loading the journal (§5.5) and running the workflow function from the top. Each ctx.* command consults the journal at its seq:

  • Recorded (its entry exists): return the recorded value without re-executing the effect.
  • Fresh (no entry at this seq): it is the live frontier — execute it and append.

ctx.run is two-phase, intent-first:

  1. Append step-begin via CAS before running the function. A fresh zombie that lost the begin-CAS is fenced here and never fires the effect.
  2. Run the function (the side effect).
  3. Append step-end (CAS). If this loses the CAS, reconcile: reload and adopt the canonical step-end value if a competitor wrote it.

Each step is journaled exactly once (the step-end CAS picks one winner); the function may execute more than once across the begin→end window (§14).

Determinism guards (§12): a command whose recorded entry has a different name at its position, a different kind at its position, or a replay that consumes fewer commands than recorded, MUST fail the run with a NonDeterminismError.

Out-of-band control. Suspension, fencing, infra-retry, and continue-as-new are not thrown through user code (a user try/catch cannot swallow them). A ctx.* operation that cannot complete inline signals the executor out of band, which maps it to a §6.4 outcome.

#11.3 Suspend / resume

sleep, await, and call suspend the run: they record their command entry, arrange a wake (a timer schedule §7, a pending signal §8, or a child start §9), and the worker acks (§6.4). A wake work item later appends the matching resolve (or signal) and re-drives from §11.2; replay reaches the same ctx.*, now finds it resolved, and continues.

#11.4 continueAsNew

ctx.continueAsNew(input) ends the current generation and atomically starts a fresh one:

  1. The worker appends a continued terminal: nextRunId = {base}_g{generation+1}, the next generation, the chain base, and the next input.
  2. It starts the next generation (§11.1: append its start with generation/base/parent carried, and enqueue) idempotently, then writes status: continued with next = nextRunId (§10).

Each generation has a fresh, empty journal, so a run may recur indefinitely with a bounded log. Code after continueAsNew never runs (it suspends like a terminal op).

#11.5 Result

To await a run's outcome, poll its journal (§5.5) for a terminal:

  • result → return its value.
  • continued → set runId = nextRunId and continue polling (the original handle transparently follows the chain to whichever generation finally returns).
  • fail → throw WorkflowFailure carrying "{error.name}: {error.message}".

Resolving against the journal (not the KV hint) is required — the journal is the truth. A bounded poll that never observes a terminal throws DurableTimeoutError (§12).

#11.6 Status

status(runId) reads the DE_RUNS record (§10). Absent record → pending. This is a hint; callers needing certainty use result (§11.5).

#11.7 Cancellation

client.cancel(runId, { reason? }) is an operational hard-stop, delivered like a signal:

  1. The client publishes a cancel work item to the run's de.work.{workflow} with msgID = {runId}:cancel (§6.2). It is a no-op on an already-terminal run, except continued (step 5).
  2. The worker, before replay, appends a cancel-requested entry (CAS, idempotent on {runId}:cancel), then re-drives the run (§11.2) — mirroring the signal / timer event-append.
  3. Deterministic injection. At the next suspend frontier (the first unresolved sleep/await/call/future-await/race), the run is cancelled iff no resolve for that frontier op has a stream-seq less than the cancel-requested entry's stream-seq (i.e. the op had not already completed). This is the same stream-seq rule that resolves signal-vs-timeout (§8) — deterministic across replays. A run with no suspend point runs to completion (a late cancel is a no-op).
  4. Uncatchable hard-stop. On cancellation the worker runs any registered ctx.onCancel handlers (LIFO, in a non-cancellable context; they MAY suspend), then appends the cancelled terminal and writes the DE_RUNS hint. The main flow is abandoned at the frontier — a user try/catch / finally in the main flow does not run (cleanup goes through onCancel).
  5. Cascade & chain. The worker publishes cancel to each in-flight child (a call with no resolve), recursively. A child cancelled independently resolves its parent's call as a failure (error.name = "CancelledError") — a normal, catchable workflow error. If the run is already continued, the worker forwards the cancel to nextRunId, so cancellation follows the chain.

Handle.result() on a cancelled run rejects with CancelledError (§12); status() reports cancelled.

#12. Error model

Error Cause Effect
FencedError A CAS append lost (10071). The losing worker stops; its work item is term'd (§6.4). Out of band — never thrown through user code.
NonDeterminismError Replay diverged from the journal (name / kind mismatch, shorter path). The run fails terminally — a fail entry is journaled (§5.3).
WorkflowFailure The workflow function threw, or a sub-workflow failed. The run fails terminally — fail journaled.
SerializationError A step result / input / continue value cannot be encoded. Deterministic programming error → the run fails terminally (it would loop forever if retried).
DurableTimeoutError A bounded wait elapsed (ctx.await timeoutMs, or result(timeout)). For await: a catchable control-flow signal. For result: a client-side timeout.
CancelledError client.cancel hard-stopped the run (or a child was cancelled). result() rejects with it; surfaced to a parent ctx.call as a catchable failure. Out of band — never thrown through main user code (§11.7).

The dividing line: a logic error (a throw, a failed assertion, replay non-determinism, an unencodable value) fails the run — the error is journaled as a terminal fail and result rejects with it. An infrastructure error (a transient publish/scheduler failure that is not a fence) is retried — the work item is nak'd and redelivered, and replay resumes safely (§6.4). A blip never loses work; only genuine logic errors end a run.

#13. Invariants

A conforming implementation MUST preserve all four:

  1. The ordered journal (de.journal.{runId}) is the sole source of truth. The KV index is a cache/hint only.
  2. Every progress decision is a CAS append or an ordered replay — never a plain mutable read (§2.2).
  3. Single-writer per run is the step-begin / append CAS — the OPTIONAL advisory run lease (a dedicated KV bucket) only cuts how often overlapping executions start; it is never the guard.
  4. The step idempotency key is {runId}:{seq} (step-begin.key, exposed as ctx.key()); dedup is a convenience, but seq + CAS is the guard.

#14. Guarantees — exactly-once state, at-least-once effects

Each step is journaled exactly once (so the run's result is deterministic), but a step's function may execute more than once.

  • Exactly-once state. The step-end CAS picks a single winner per seq; replay returns recorded values. No matter how many workers, crashes, or redeliveries a run took, its result is built from the recorded entries.
  • At-least-once effects. Between step-begin and step-end the result is not yet recorded, so a resume (crash, ack-timeout redelivery, or a second worker on a concurrent wake) re-runs the function. A fresh zombie is fenced before the effect (§11.2), but an in-flight one can repeat.
  • Making effects safe. A non-idempotent side effect SHOULD key on ctx.key() ({runId}:{seq}, stable across every re-run of that step) — e.g. INSERT … ON CONFLICT (key) or a provider-side idempotency key. This is the rule in any multi-worker deployment, not a crash-only precaution.

#15. Versioning

  • The wire format is v:1, frozen. Every entry carries "v": 1; decoders MUST reject other versions (§5.1). A wire-breaking change bumps v; there is no in-band v:1↔v:2 shim.
  • The library (@synadia-ai/durable) is independently versioned (pre-1.0, unstable). It may add capabilities without changing the wire (cancellation, version-pinned dispatch, and the gating run lease all arrived this way); such additions are additive and interoperable with existing v:1 journals.
  • The workflow version (wfVersion on the start entry) is the dispatch key. A worker MAY register multiple (name, version) pairs and MUST resolve the function by the run's recorded wfVersion, executing that exact version. A (name, version) is immutable code. If the recorded version is not registered, the worker MUST park the run (status blocked) and retry until it is deployed, never failing or dropping it. continueAsNew MAY carry a target version (the migration path). In-code patching is not in wire v1.

#16. Implementation checklist

A worker conforms when it:

  • Provisions the resources with the §3 config (idempotently), under a shared prefix.
  • Binds, per workflow, a de-worker-{workflow} consumer on DE_WORK and a de-timer-{workflow} consumer on DE_TIMERS (Explicit ack, max_deliver: -1) (§6.3, §7).
  • Drives each work item by loading the journal and executing two-phase, intent-first, with the determinism guards (§11.2); records signal / resolve events before replay (§7–§9).
  • Appends entries with the §5.4 msgID + expect CAS; treats 10071 as a fence and terms (§6.4).
  • Acks on terminal / suspend / continue, naks on infra error, terms on fence / unparseable / no-journal (§6.4).
  • Arms timers via native scheduling to a same-stream wake subject, deduped by {runId}:sched:{seq} (§7).
  • Starts sub-workflows with a parent link and delivers child-done to the parent's queue on terminal (§9).
  • Writes the DE_RUNS hint without ever downgrading a terminal status (§10).
  • Fails the run terminally on a logic error (throw / non-determinism / unencodable) and retries (nak) on a non-fenced infra error (§12).

A client conforms when it:

  • Validates identifiers (§4.2) and starts runs idempotently ({runId}:start + expect: 0) (§11.1).
  • Resolves result against the journal terminal and follows continued across generations (§11.5).
  • Sends signals as signal work items to the run's workflow queue, with a unique id (§8).
  • Treats DE_RUNS (status) as a hint, not as truth (§10).

#Appendix A — Resource & subject quick reference

# Streams / KV (prefix "de"; uppercase for names)
DE_JOURNAL   stream  file/limits             subjects: de.journal.*
DE_WORK      stream  file/workqueue          subjects: de.work.*
DE_TIMERS    stream  file/limits + schedules subjects: de.timer.>
DE_RUNS      kv      file, history 1         key: {runId}
DE_LEASES    kv      file, history 1, ttl    key: {runId}     (optional gating lease)

# Subjects
de.journal.{runId}                 # the run's journal (truth)
de.work.{workflow}                 # the workflow's work queue
de.timer.sched.{runId}.{seq}       # a timer's schedule definition
de.timer.wake.{workflow}           # fired-timer wake delivery

# Consumers (per workflow)
de-worker-{workflow}   on DE_WORK    filter de.work.{workflow}
de-timer-{workflow}    on DE_TIMERS  filter de.timer.wake.{workflow}

# Identifier rule: ^[A-Za-z0-9_-]+$   (subject token ∩ KV key)
# Derived runIds: continue-as-new {base}_g{n} (gen 0 = base);  child {parentRunId}_{seq}

#Appendix B — msgID conventions

# Journal entries (on de.journal.{runId})
{runId}:start            # start
{runId}:{seq}:b          # step-begin
{runId}:{seq}:e          # step-end
{runId}:{seq}            # now / random / uuid / sleep / await / call
{runId}:sig:{signalId}   # signal
{runId}:{seq}:r          # resolve
{runId}:cancel           # cancel-requested
{runId}:terminal         # result / fail / continued / cancelled  (one terminal per run)

# Work items (on de.work.{workflow})
{runId}:work:start       # start item
{runId}:sig:{id}         # signal item
{childRunId}:done        # child-done item (to parent's queue)

# Timer schedule (on de.timer.sched.{runId}.{seq})
{runId}:sched:{seq}      # the native schedule (dedups re-arm)
