How it works

A plain-language tour: no durable-execution server, the journal-then-replay trick, and the one NATS primitive that makes it correct.

A friendly tour of @synadia-ai/durable — what it is, the handful of ideas it rests on, and which NATS features do the work. If you want the rigorous version with exact signatures and invariants, read DESIGN.md. This page is the gentle one.

#1. The big idea: there is no durable-execution server

Most durable-execution systems ship a server — a dedicated control plane that stores your workflow state, holds the queues and timers, and drives execution. You run it, scale it, secure it, and back it with a database.

We don't have one. The NATS cluster is the engine. This library is just code that runs inside your own processes; it uses JetStream streams and KV — things NATS already does — as the durable substrate.

   TRADITIONAL (Temporal, Resonate, …)             @synadia-ai/durable
   ───────────────────────────────────             ────────────────────────────

   ┌───────────┐      ┌───────────────────┐         ┌───────────┐      ┌────────────────────┐
   │ your app  │─────▶│  DURABLE-EXEC     │         │ your app  │─────▶│   NATS cluster     │
   │ + workers │◀─────│  SERVER           │         │ + workers │◀────▶│   (you already     │
   └───────────┘      │  state · queues · │         └───────────┘      │    run this)       │
                      │  timers · sched.  │                            └────────────────────┘
                      └────────┬──────────┘            a library turns NATS streams + KV into
                               │ persists to           the journal, queue, scheduler & index.
                        ┌──────▼───────┐                       NO new service to operate.
                        │  Postgres /  │
                        │  Cassandra   │
                        └──────────────┘

So "no server" doesn't mean "no infrastructure" — NATS is infrastructure. It means no new, durable-execution-specific service. If you already run NATS, you already run the engine.

#2. What durable execution buys you

Imagine a function that reserves inventory, charges a card, and ships an order. If the process crashes after charging but before shipping, a naïve retry double-charges, and a non-retry loses the order. You end up hand-rolling state machines, status columns, and idempotency tables.

Durable execution makes that function survive crashes on its own. You write normal code; the runtime guarantees that on a restart it picks up exactly where it left off — already-completed steps are not run again, and the unfinished one resumes.

   reserve ✓ ─▶ charge ✓ ─▶ ✗ CRASH ─▶ … restart … ─▶ reserve (skipped) ─▶ charge (skipped) ─▶ ship ✓

#3. The core trick: journal, then replay

A workflow is an ordinary async function. Every step that touches the outside world or is otherwise nondeterministic goes through ctx — and ctx writes the step's result to an ordered journal before returning it.

When a worker (re)starts a run, it does not resume a paused coroutine. It simply re-runs the function from the top — but now each ctx.* call first checks the journal:

  • already recorded? → return the saved value, don't run it again (this is replay).
  • not yet recorded? → this is the live frontier: actually run it, record the result, continue.
   FIRST EXECUTION                         REPLAY AFTER RESTART
   ──────────────                          ────────────────────
   await ctx.run("reserve") ─▶ runs → 📓   await ctx.run("reserve") ─▶ 📓 saved value (no re-run)
   await ctx.run("charge")  ─▶ runs → 📓   await ctx.run("charge")  ─▶ 📓 saved value (no re-run)
   await ctx.run("ship")    ─▶ runs → 📓   await ctx.run("ship")    ─▶ runs for real  ◀─ frontier
            ✗ crashed here                          resumes precisely here

The one rule this asks of you: everything nondeterministic goes through ctx (no bare Date.now(), Math.random(), or I/O in the workflow body). The boundary is easy to see — it's literally the ctx. prefix — and the runtime fails loud if replay ever diverges.

#4. The primitives (and the NATS feature behind each)

What you write, what it means, and what NATS feature implements it:

You write What it gives you NATS feature behind it
ctx.run(name, fn) a durable step append to the JetStream journal stream (two-phase, CAS-fenced)
ctx.now() / random() / uuid() deterministic values journaled value-capture (a stream append)
ctx.sleep(name, ms) a durable timer native JetStream message scheduling (server delivers a wake)
ctx.await(name, {timeoutMs?}) wait for a signal (optional timer) a journaled signal in arrival order; a timeout races a durable timer
ctx.call(wf, input) invoke a sub-workflow a child run + a work-queue item
ctx.startCall / startSleep / startAwait start an op without blocking → a DurableFuture the same call/sleep/await commands — no new wire kinds
await Promise.all([...]) / ctx.race([...]) fan out & await many ops concurrently each future resolved from the journal; race picks the journal-earliest branch
ctx.continueAsNew(input) restart fresh; bound the journal a continued terminal → a new generation (chained runId)
ctx.onCancel(fn) compensation that runs if the run is cancelled journaled cleanup steps, run LIFO & non-cancellably
ctx.key() an idempotency key for your effect derived from runId:stepSeq
client.start / result / status launch & observe a run stream append (start) + reading the journal / KV
client.cancel(runId) hard-stop a run → terminal cancelled a cancel work-item → cancel-requested + cancelled journal entries
a worker runs your workflow code a JetStream pull consumer (queue group) over the work stream

Every primitive above is implemented and tested today — the full ctx surface, fan-out and cancellation included.

The whole thing rests on one NATS capability, used everywhere (see §7): an optimistic append (expect.lastSubjectSequence) plus message dedup (Nats-Msg-Id).

#5. The moving parts

A small, fixed footprint lives on NATS. Your workers and clients are just the library:

   ┌──────────────────────────── NATS cluster — "the engine" ─────────────────────────────┐
   │                                                                                      │
   │   DE_JOURNAL  (stream)        DE_WORK  (work queue)         DE_RUNS  (KV)            │
   │   subject: de.journal.{run}   subject: de.work.{workflow}   key: {run} → status      │
   │   the ordered SOURCE OF       "go make progress on X"       an INDEX / cache         │
   │   TRUTH, one log per run      items, load-balanced          (status & result hint)   │
   └─────▲─────────────▲────────────────────▲───────────────────────────────▲─────────────┘
         │ append &    │ read               │ pull (shared queue group)     │ read/write
         │ replay      │                    │                               │
   ┌─────┴─────────────┴────┐        ┌──────┴────────────────┐       ┌──────┴────────────┐
   │  CLIENT                │        │  WORKER(s)            │       │  handle.status()  │
   │  client.start(wf, in)  │        │  pull → load journal  │       └───────────────────┘
   │  handle.result()       │        │  → replay → advance   │
   └────────────────────────┘        │  → ack / complete     │
                                     └───────────────────────┘
  • DE_JOURNAL — the truth. One ordered log per run (one NATS subject per run, not one stream per run, so it scales to many runs without hitting stream-count limits).
  • DE_WORK — a JetStream work queue. A "start", "resume", or "signal" item lands here; whichever worker is free pulls it. The queue load-balances; the journal keeps the truth. (Timer wakes ride a parallel DE_TIMERS stream populated by native scheduling — see §7.)
  • DE_RUNS — a KV bucket used only as a fast status index. It is never trusted for a progress decision (those go to the journal).

#6. The life of a run

   client.start(charge, input)
        │  1. append the "start" entry to DE_JOURNAL  (idempotent on runId)
        │  2. drop a work item on DE_WORK
        ▼
   ┌─ a worker pulls the item ─────────────────────────────────────────────────┐
   │   3. load the run's journal from DE_JOURNAL                               │
   │   4. re-run the workflow function from the top:                           │
   │        • journaled steps  → return saved values (replay)                  │
   │        • first new step   → run it, append the result (the frontier)      │
   │   5a. reached the end   → append "result", update DE_RUNS, ACK the item   │
   │   5b. hit a sleep/await  → suspend; a wake re-enqueues a work item later  │
   │   5c. worker died here   → the item is redelivered → another worker       │
   │                            loads the journal and resumes (back to step 3) │
   └───────────────────────────────────────────────────────────────────────────┘

   handle.result()  ─▶ resolves when the journal gets its terminal "result" entry

Because step 4 always rebuilds state from the journal, a crash anywhere is just "go back to step 3."

#7. What makes it correct: one primitive, one error code

The subtle part of any durable system is: when a worker is slow (not dead) and its work gets redelivered, two workers can briefly run the same run. If both charge the card, you've lost.

NATS gives us exactly the right tool: an optimistic append. When you publish to a run's journal subject, you can say "only accept this if the subject's last entry is still sequence S." If someone else already appended, the server rejects you with one specific error — 10071, "wrong last sequence." That single mechanism is the whole correctness story (and KV CAS — create / update(rev) — is the same mechanism under the hood, since KV is itself a stream).

We use it to make the side effect fire at most once in the common case by recording a step's intent before running it (this is the "two-phase" step):

   Two workers both reach step N (slow worker B was redelivered the same run):

     worker A ─▶ append  step-begin(N)  ┐
                                        ├──▶ NATS accepts only ONE (expect last seq = …)
     worker B ─▶ append  step-begin(N)  ┘        A wins  ·  B gets error 10071

         A: runs the side effect ─▶ append step-end(N) ─▶ continues
         B: a FRESH zombie is fenced BEFORE running anything ─▶ stops quietly

So the queue does not decide who may write — the log does, and each step is journaled exactly once (so the workflow's result is deterministic no matter how many workers touched the run). What's at-least-once is the step fn's execution: if a worker is mid-step (begin written, end not yet) and the run is also driven elsewhere — a crash + resume, an ack-timeout redelivery, or a second worker that picked up a concurrent wake — the resume path re-runs fn. That's what the idempotency key (ctx.key()) is for. So: exactly-once state, at-least-once effects — the standard durable-execution contract. (In practice the window is kept rare: ack-progress heartbeats stop mid-step redelivery of long steps, and a default-on advisory run lease stops most overlapping executions from starting — both fail open, so correctness never depends on them.)

Two more safety nets, both free from NATS:

  • Dedup (Nats-Msg-Id) — a redelivered identical append is recognized, not duplicated.
  • Fail loud — if a replay ever produces a different step than the journal recorded (nondeterministic code), the run aborts with a clear error instead of silently corrupting.

#8. Why "no server" is a big deal

  • Nothing new to operate. No DE server to deploy, scale, patch, secure, or pay for — and no separate database. Your workers are stateless; all durability lives in NATS.
  • It rides your existing NATS. Multi-region, clustering, auth, back-pressure, and observability are whatever your NATS already provides.
  • It's a library. Embed it; there's no network hop to a control plane to start a workflow.
  • Portable by design. The on-stream format is language-neutral, so Go / Python / … workers can share the same runs later.

#9. Status & where to go next

Built and tested today — the full ctx surface: ctx.run (two-phase, crash-safe), ctx.now/random/uuid, ctx.sleep durable timers (native JetStream scheduling), ctx.await signals (client.signal) + timeoutMs (a durable timer races the signal), ctx.call sub-workflows, ctx.continueAsNew (chained generations → bounded journals), durable fan-out (ctx.startCall/startSleep/startAwait → a DurableFuture; native Promise.all/Promise.allSettled; ctx.race), cancellation (client.cancel → an uncatchable hard-stop → terminal cancelled, opt-in ctx.onCancel compensation + child cascade), the client (start / result / status), the worker pull-replay loop (bounded concurrency, supervised consumers, ack-progress heartbeats for long steps, a default-on advisory run lease, an injectable metrics/health seam), large-payload offload (values over the broker limit ride the DE_BLOBS object store as digest-verified refs), and self-reconciling resource provisioning — all green (314 tests), with a suite proving replay-without-re-execution, crash-resume, fence-before-effect, real-timer suspend/resume, signal FIFO/early delivery, sub-workflow calls, continue-as-new chains, durable fan-out (partial-completion replay + deterministic ctx.race), cancellation (uncatchable hard-stop, onCancel compensation, child cascade, cancel-vs-completion by stream-seq), a multi-worker concurrency/chaos pass (exactly-once journaling, at-least-once effects — and exactly-once execution with the lease on), large-payload offload (multi-MiB values round-tripping, crash-replay from the blob store, park-and-heal on a missing blob), and the NATS primitives.

Versioning. A worker can register several versions of a workflow (defineWorkflow(name, fn, { version })); each run stays on the version it started under, so shipping new code never breaks runs already in flight. Migrate a long-running one forward with ctx.continueAsNew(input, { version }). A run whose version isn't deployed anywhere waits (blocked) instead of failing.

A payload bigger than the server's message limit (a model output, a scraped page, a file) doesn't fail the run: it is offloaded — the value is written to a DE_BLOBS object store and the journal records a small content-addressed, digest-verified reference instead; replay fetches and verifies it transparently. On by default; offload: false restores the strict "fail loud on oversized" behavior.

Not built yet: in-code patching (patched()), automatic GC of finished generations, suspending ctx.onCancel cleanup, and ctx.startRun local-step fan-out (see DESIGN.md §9).
