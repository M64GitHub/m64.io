Deep-dive design

The rigorous account: exact API surface, the two-phase replay algorithm, the resource layout, and every invariant the correctness model rests on.

Status: implemented & evolving. This began as the Phase 0 design note; the library now implements the full ctx surface (run / now / random / uuid / sleep / await / call) plus the client and worker. The model, invariants, and resource layout below are kept current. A few API points have shifted since (e.g. ctx.assert returns void, not an asserts predicate; there is no gating lease yet — the journal CAS is the single-writer guard); the code is the source of truth for exact signatures, CHANGELOG.md tracks what's shipped, and LEARNINGS.md records the gotchas.

Lightweight durable execution implemented entirely on NATS/JetStream primitives. The NATS cluster is the engine; this is a library, not a server. Guiding principle (your steer): maximum correctness over brief-adherence — when a more robust path exists, take it.

Target substrate: Synadia Cloud (NGS) first. Verified live against your synadia-cloud account: NGS server 2.14.2, tier R1, 8 MiB max payload, streams require max_bytes, file storage only (no memory tier). Built on @nats-io/{nats-core,jetstream,kv,transport-node} 3.4.0. Every primitive this design leans on was empirically confirmed (Appendix A).

#0. TL;DR — the one-primitive insight

Everything that must be correct under concurrency reduces to a single NATS primitive:

Optimistic append: publish with expect.lastSubjectSequence = S. The server accepts only if the named subject's last message is still at sequence S, else rejects with JetStreamApiError code 10071 (StreamWrongLastSequence).

  • The journal is an ordered, append-only log per run. Appending with expect.lastSubjectSequence makes it structurally single-writer: two workers racing → exactly one wins, the other gets 10071 and knows it is a zombie.
  • The KV lease is the same mechanism: kv.update(key, val, rev) is expect.lastSubjectSequence = rev on the key's subject; kv.create is first-writer-wins.
  • Side effects are intent-first (two-phase): we CAS a step-begin before running the effect, so a redelivered zombie is fenced before it can fire the effect (§4, §6).

One error code (10071) is caught everywhere a writer loses a race; progress is never gated on a plain read.

#1. Programming model

A workflow is a normal async function. Every nondeterministic or side-effecting operation goes through ctx.*, which journals to the ordered log. On crash/restart the function re-runs from the top; journaled ops return their recorded value (replay, no re-execution); the first un-journaled op is the live frontier and actually runs. ctx.run calls are blocking; start an op without blocking with ctx.startCall → a DurableFuture, and fan out with Promise.all / ctx.race.

ts
const charge = defineWorkflow(
  "charge",
  async (ctx: Ctx, input: { cartId: string }): Promise<Receipt> => {
    const order = await ctx.run("reserve", () => reserveInventory(input.cartId));
    const approved = await ctx.await<{ ok: boolean }>("payment"); // external signal
    if (!approved.ok) return { status: "abandoned" };
    await ctx.sleep("settle-delay", 2_000); // durable timer (native schedule)
    const ship = await ctx.call(shipWorkflow, { orderId: order.id }); // sub-workflow
    return { status: "shipped", tracking: ship.tracking };
  },
);

Determinism boundary = ctx. Code outside ctx.* must be pure/deterministic (no Date.now(), Math.random(), I/O, ambient mutable state). We fail loud on detected divergence (§6), per Tiger-Style. We use async/await, not generators (Appendix B).

#2.2 The context object

ts
interface Ctx {
  readonly runId: string;
  readonly workflow: string;
  readonly version: number;
  readonly attempt: number; // redelivery attempt of the current frontier (informational)

  // --- core primitive: journal a step (intent-first, two-phase under the hood) ---
  run<T>(name: string, fn: () => Promise<T> | T, opts?: StepOptions): Promise<T>;

  // --- deterministic builtins (single journaled value-capture each) ---
  now(): Promise<number>; // ms since epoch, recorded at frontier
  random(): Promise<number>; // [0,1), recorded
  uuid(): Promise<string>; // recorded

  // --- durable timer: suspend until due (native JetStream schedule) ---
  sleep(name: string, ms: number): Promise<void>;

  // --- signal: block until externally resolved (durable-promise-as-signal) ---
  await<T>(name: string, opts?: { timeoutMs?: number }): Promise<T>;

  // --- sub-workflow ---
  call<I, O>(workflow: Workflow<I, O>, input: I, opts?: CallOptions): Promise<O>;

  // --- end this generation, start a fresh one (bounds the journal for long/looping runs) ---
  continueAsNew(input: unknown): Promise<never>; // code after never runs; result() follows the chain

  // --- durable fan-out: start an op without blocking → a DurableFuture (a thenable) ---
  //     await f / Promise.all / Promise.allSettled all work; race with ctx.race (native Promise.race is unsafe).
  startCall<I, O>(workflow: Workflow<I, O>, input: I, opts?: CallOptions): DurableFuture<O>;
  startSleep(name: string, ms: number): DurableFuture<void>;
  startAwait<T>(name: string, opts?: { timeoutMs?: number }): DurableFuture<T>;
  race<F extends DurableFuture<unknown>>(futures: F[]): Promise<Awaited<F>>; // journal-earliest wins

  // --- cancellation: opt-in compensation; runs LIFO & non-cancellably before the cancelled terminal ---
  onCancel(fn: () => void | Promise<unknown>): void;

  // --- step idempotency key for user side-effects: `${runId}:${stepSeq}` (invariant #4) ---
  key(): string;

  // --- dependency injection (NATS conn, http clients, config) ---
  get<T>(key: string): T;

  // --- invariant guard: fail loud, abort the run (do not catch) ---
  assert(cond: boolean, msg?: string): void;
}

interface StepOptions {
  retry?: RetryPolicy; // inline retry of fn before journaling failure; default: none
}
type RetryPolicy =
  | { kind: "none" }
  | { kind: "fixed"; attempts: number; delayMs: number }
  | { kind: "exponential"; attempts: number; baseMs: number; factor?: number; maxMs?: number };

interface CallOptions {
  runId?: string;
} // default child runId is deterministic: `${parent}_${seq}` (an underscore — runIds are subject tokens)

/** Opaque handle to a durable op started but not awaited (`ctx.startCall/startSleep/startAwait`). A thenable —
 *  `await f`, `Promise.all([...])`, `Promise.allSettled([...])` all work; race with `ctx.race`. */
type DurableFuture<T> = PromiseLike<T>;

now/random/uuid are Promise-returning because the recorded value is durably appended on first execution (mental model: anything through ctx is awaited). now() cannot be seed-derived, so per-call journaling is the only correct option (Q2).

#2.3 Client (start / observe / signal)

  state: "pending" | "running" | "suspended" | "completed" | "failed" | "continued" | "cancelled";

#3. The journal (language-neutral wire layout)

One ordered log per run on subject de.journal.{runId} in stream DE_JOURNAL. Append-only; the sole source of truth (invariant #1). Each message is one entry:

jsonc
{
  "v": 1,                       // wire-format version (frozen contract for Go/Python/Zig SDKs)
  "seq": 3,                     // command position, monotonic per run (0,1,2,…)
  "kind": "step-begin",         // start|step-begin|step-end|now|random|uuid|sleep|await|signal|call|resolve|result|fail|continued|cancel-requested|cancelled
  "name": "reserve",            // user label (step/sleep/await/call); asserted on replay
  "ts": 1718563200000,          // wall clock at write (informational only)

  // kind-specific:
  "wf": "charge", "wfVersion": 1, "input": {…},   // start (also: generation, base — continue-as-new chain)
  "key": "r_ab:3",                                 // step-begin (idempotency key for the side-effect)
  "value": { "id": "o_9" },                        // step-end / now / random / uuid / resolve
  "dueMs": 1718563260000, // sleep (absolute deadline, deterministic)
  "childRunId": "r_ab_3",                          // call (underscore: runIds are single subject tokens)
  "data": {…},                                     // signal (inbound, matched to an await by name+order)
  "resolvesSeq": 3,                                // resolve (timer/child completion for command seq 3)
  "nextRunId": "r_ab_g1"                           // continued (terminal: next generation = `${base}_g${n}`)
}

seq is the identity of a command (positional, Resonate-style). Two-phase steps: step-begin and step-end share the same seq (the begin is the intent/fence, the end is the result). Pure captures (now/random/uuid) are a single entry. signal and resolve do not consume a new seq — signal is matched to an await by name+arrival-order; resolve references resolvesSeq.

msgID (dedup) + expect.lastSubjectSequence (CAS) on every append:

entry msgID fences against
start {run}:start duplicate starts
step-begin (seq N) {run}:{N}:b double-fire: a zombie loses here, before the side effect
step-end (seq N) {run}:{N}:e duplicate result after ack-loss / concurrent resume
now/random/uuid/sleep/await/call (seq N) {run}:{N} double-advance
signal {run}:sig:{signalId} duplicate signal delivery
resolve (seq N) {run}:{N}:r duplicate timer/child completion
cancel-requested (non-command) {run}:cancel duplicate cancel delivery (double-cancel collapses to one)
result / fail / continued / cancelled {run}:terminal double-finalize (exactly one terminal per generation)

Dedup is a convenience (bounded duplicate_window); the seq + CAS is the real exactly-once guard (invariant #4).

#4. Replay algorithm (two-phase, intent-first)

executeRun(workItem):                         # workItem = {runId, workflow, kind, payload?}
  lease = acquireOrRenewLease(runId)          # designed coarse gate — this design SKIPS it; the journal CAS below is the sole guard
  journal = loadJournal(runId)                # ordered read of de.journal.{runId}
  if workItem carries an unjournaled event (signal/timer/child-done):
      appendEvent(runId, event)               # single-writer CAS append (idempotent via msgID)
  ctx = new Ctx(runId, journal, lease); ctx.seq = 0; ctx.frontier = highestCommandSeq(journal)+1
  try:
      out = await workflowFn(ctx, journal.start.input)
      appendTerminal(runId, {kind:"result", value: out})   # CAS; TRUTH
      writeRunIndex(runId, "completed", out)                # KV cache
      ack(workItem); releaseLease(lease)
  catch Suspend:                              # sleep/await/call not yet resolvable
      ack(workItem); releaseLease(lease)      # a wake (timer/signal/child-done) re-enqueues
  catch Fenced(10071):                        # lost a CAS race → zombie
      term(workItem)                          # another worker owns this run; stop silently
  catch WorkflowError(e):
      appendTerminal(runId, {kind:"fail", error: e}); writeRunIndex(runId, "failed", e); ack(workItem)

# the core step primitive — note the begin-CAS happens BEFORE the side effect:
async run(name, fn, opts):
  N = this.seq++
  st = this.stepState(N)                      # {begin?, end?} from the loaded journal
  if st.end:                                  # ---- REPLAY (completed) ----
      assert(st.begin.name == name, "non-determinism at seq "+N)   # FAIL LOUD
      return st.end.value                     # fn is NOT executed
  if st.begin and not st.end:                 # ---- RESUME (crashed mid-step) ----
      assert(st.begin.name == name)
      result = await withRetry(opts.retry, fn)         # at-least-once (idempotency key st.begin.key)
      appendStepEnd(runId, N, result)                  # CAS; on 10071 adopt the winner's end
      return result
  else:                                       # ---- FRONTIER (fresh) ----
      appendStepBegin(runId, N, name, key=`${runId}:${N}`)  # CAS → throw Fenced if a zombie; NO effect yet
      result = await withRetry(opts.retry, fn)              # side effect runs exactly here, once
      appendStepEnd(runId, N, result)                       # CAS
      this.frontier = N+1
      return result

now/random/uuid are a single value-capture (generate → CAS-append → return; replay returns recorded). sleep/await/call register a command then arrange a wake and throw Suspend unless already resolvable; replay returns the resolved value if a matching resolve/signal exists, else throw Suspend (§7).

#5. Resource layout (NGS-first)

Small fixed footprint per deployment (prefix configurable, default de). NGS constraints baked in: every stream sets max_bytes; file storage; entries ≤ 8 MiB.

resource kind subjects / keys retention / config role
DE_JOURNAL stream (file) de.journal.* (token = runId) limits; max_bytes source of truth — ordered per-run log
DE_WORK stream (file) de.work.* (token = workflow) work-queue; max_bytes unit-of-progress items; one durable pull consumer per workflow, shared across workers (queue-group load-balancing); AckWait, MaxDeliver>1
DE_TIMERS stream (file) de.timer.> (sched + wake) limits; max_bytes; allow_msg_schedules @at schedule definitions + same-stream wake delivery (cross-stream targets rejected); one timers consumer per workflow
DE_RUNS KV key = runId → {status, wf, version, result?/error?} history 1 index/cache (status hint); never gates progress. A gating lease is future work (§6, §10).

One journal stream for all runs (per-run subject, not per-run stream) → avoids NGS stream-count limits. One shared pull consumer per workflow type, never per run.

Work items + timers. DE_WORK carries start, resume, signal(+data), and child-done(+result) — one durable queue for dispatch and signal delivery, so signals need no extra stream (they reuse the work queue). Timers are the exception: the server rejects a schedule whose target is in a different stream, so ctx.sleep uses the dedicated DE_TIMERS stream, which holds both the @at schedule (de.timer.sched.{run}.{seq}) and its wake subject (de.timer.wake.{wf}); a per-workflow timers consumer pulls the wakes. (Alternative if signals grow large/high-volume: split out DE_SIGNALS — Q4, deferred.)

Reads never trust a follower. KV get may be replica-served (Bucket.direct is real), so progress decisions use CAS or ordered journal replay — never a plain get. status() reads the KV index as an explicit hint; result() resolves on the journal's terminal entry (truth).

Payload bound. Inputs/results/step-values share an 8 MiB ceiling on NGS. The design documents the limit; large blobs (object-store offload via @nats-io/obj) are a v-next hook.

#6. Invariant enforcement (the correctness contract)

# Invariant Mechanism (verified)
1 Ordered journal = sole truth; KV = index/cache All decisions derive from de.journal.{run} (ordered). KV DE_RUNS is written after the journal, read only as a hint. result() resolves on the terminal journal entry, not KV.
2 Every progress gate via CAS-on-revision or replay — never plain get All appends use expect.lastSubjectSequence; lease take-over/renew uses kv.update(rev). Both leader-linearizable; both fail 10071. No path gates on kv.get.
3 Single-writer per run (the journal CAS); assume a slow (not just dead) worker is redelivered The guard (built): the step-begin append is CAS and precedes the side effect — a redelivered slow/zombie worker loses step-begin with 10071 and aborts before firing the effect. The queue never authorizes a write; the log does.
4 Step idempotency key {run}:{step}; dedup is convenience ctx.key() = {runId}:{stepSeq}, recorded in step-begin and used as the effect's idempotency key. Real exactly-once = positional seq + CAS; duplicate_window dedup is a bounded convenience on top.

Empirically proven (Appendix A): concurrent appends → exactly one wins; concurrent lease renews → exactly one wins; stale revision → rejected; all losers surface 10071.

#7. Crash semantics, timers, signals, sub-workflows

Exactly-once boundary. A completed step (step-begin+step-end) replays exactly-once — fn is never re-run — and every step is journaled exactly once: the step-end CAS lets only one writer win, and a loser reconciles by adopting the recorded value (so the workflow result is deterministic no matter how many workers touched the run). Because step-begin is CAS-fenced before the effect, a fresh zombie is fenced before it can fire fn. What is at-least-once is fn execution across the begin→end window: if a worker is mid-step (begin written, end not yet) and the run is also driven elsewhere — a crash + resume, an AckWait redelivery, or a second worker processing a concurrent wake — the resume path re-runs fn. There is no gating lease, so in a multi-worker deployment this window is real (not only on crash). ctx.key() (${runId}:${seq}, stable across re-runs) is the idempotency key that makes the user's side effect safe, and a gating lease would narrow (not remove) the window. This is the standard durable-execution contract — exactly-once state, at-least-once effects — and is strictly stronger than run-then-journal-result. Proven in the multi-worker chaos test (exactly-once journaling, at-least-once execution).

Timers (ctx.sleep). Frontier: append {kind:"sleep", dueMs}, then publish a native scheduled message to the dedicated DE_TIMERS stream (schedule:{ specification:"@at <iso>", target:"de.timer.wake.<wf>" }) so the server delivers a timer work-item at dueMs; suspend. The schedule and its target must live in the same stream (cross-stream targets are rejected) — hence DE_TIMERS holds both de.timer.sched.{run}.{seq} and de.timer.wake.<wf>, and the worker pulls wakes via a per-workflow timers consumer. Wake → worker appends {kind:"resolve", resolvesSeq:N} and replays. On a replay before the timer fires, the schedule is re-ensured idempotently (msgID dedup) so a crash between journaling the command and scheduling can't deadlock. Verified: delivery at +1504 ms for a +1500 ms request (Appendix A). Portable fallback (older servers): a leader-elected scanner over a timer KV — documented, not built yet.

Signals (ctx.await). client.signal(run, name, data) publishes a signal work-item. The owning worker appends {kind:"signal", name, data} to the journal in arrival order (single-writer). ctx.await(name) consumes the earliest unconsumed signal with that name; else throw Suspend. Early signals (before the await) sit in the log until consumed — handled naturally.

Sub-workflows (ctx.call). Frontier: deterministic childRunId = {parent}_{seq} (an underscore — runIds must be single subject tokens), append {kind:"call", childRunId, wf, input}, start the child (its start entry records a parent link {runId, seq, wf}), suspend. When the child reaches a terminal, the worker reads that parent link and emits a child-done work-item to the parent's workflow consumer; the parent appends {kind:"resolve", resolvesSeq:N} — with the child's value, or an error if the child failed (which fails the parent) — and continues. Both the child start and the child-done are idempotent (msgID) and re-emitted on redelivery, so a crash can't lose the linkage.

Continue-as-new (ctx.continueAsNew). Bounds the journal for long/looping runs: each generation is its own bounded log; a generation ends not with result but with a terminal {kind:"continued", nextRunId, input, generation, base}, and the next generation begins with a fresh start. The next-gen id is deterministic — ${base}_g${n} (the _g is KV-key-safe and distinguishable from ctx.call's _{seq} children — a runId is also a DE_RUNS KV key, which rejects characters subjects allow such as ~) — so racing/replaying workers compute the identical id without coordination. The continued terminal is executor-driven (written after the same fully-consumed divergence guard as a normal return, so a shorter replay path fails loud) and shares the {run}:terminal slot, so it races safely against result/fail (and is verified to actually land before a continue proceeds). The worker then starts the next generation idempotently, always re-publishing the start work-item (idempotent msgID) so a crash between the continued write and the enqueue can't orphan the chain; on redelivery the terminal short-circuit re-drives it. Handle.result() transparently follows continued → nextRunId to the generation that finally returns. A sub-workflow may continue-as-new — the parent link threads forward, so only the final generation notifies the parent. Old generations' logs remain for audit (GC is the documented bound).

Durable fan-out (ctx.startCall/startSleep/startAwait). A start* op journals its call/sleep/await command and arms its side effect, but returns a DurableFuture instead of suspending — so several can be in flight at once. The engine serializes their CAS appends behind a per-run append-queue (concurrent commands chain on lastSeq rather than self-fencing), and an AsyncLocalStorage step-scope keeps ctx.key() and the nesting guard correct under concurrency. await Promise.all([...]) suspends once until every branch's resolve is journaled; ctx.race([...]) resolves the journal-earliest branch by stream-seq (deterministic across replays — native Promise.race is unsafe, since a pending branch ordered first would suspend before a ready one could win). Each branch replays from the journal, so fan-out is crash-safe: a child that finished before a crash is never re-run. No new wire kinds — fan-out reuses call/sleep/await.

Cancellation (client.cancel → ctx.onCancel). client.cancel(run, {reason?}) publishes a cancel work-item; the worker appends a cancel-requested entry (idempotent on {run}:cancel) before replay, exactly like a signal. At the next suspend frontier the run is hard-stopped — an uncatchable halt surfaced out of band, so a user try/catch/finally in the main flow can't swallow it. The cancel-vs-completion race is resolved by stream-seq: a resolve/signal counts only if it landed before the cancel-requested (the same beforeCancel rule as signal-vs-timeout), so a run with no pending frontier completes and a late cancel is a no-op. Opt-in ctx.onCancel handlers then run (LIFO, in a non-cancellable context, exactly-once on replay — their ctx.run steps journal) before the terminal {kind:"cancelled", reason?}. Cancellation cascades to each in-flight ctx.call child (recursively) and follows a continue-as-new chain; a cancelled child surfaces to its parent's ctx.call as a catchable CancelledError.

Redelivery is safe by construction. There is no gating lease, and does not yet extend AckWait via working() for long steps. If AckWait lapses and a work item is redelivered, correctness holds anyway: the step-begin CAS fences the double-writer before any effect, and infra errors NAK rather than fail. A gating KV lease (to cut wasted concurrent work) and working() for long steps are documented future refinements.

#9. Non-goals (hooks left)

  • In-flight code change — shipped via version-pinned dispatch: a worker registers multiple (name, version); each run executes the version its start recorded, and continueAsNew(input, { version }) migrates a chain. A (name, version) is immutable code (mutate one → fail-loud NonDeterminismError); a run whose version isn't deployed parks (blocked). In-code patching (patched()/getVersion(), evolving an in-flight run in place) remains future.
  • Automatic journal GC / snapshotting — ctx.continueAsNew (implemented, §7) bounds per-generation journal growth; reclaiming finished generations' logs (a TTL/GC policy) is still manual.
  • Sagas / cross-workflow transactions as built-ins.
  • Suspending ctx.onCancel cleanup and ctx.startRun local-step fan-out — designed, deferred (cancellation and durable fan-out themselves are implemented, §7).
  • Object-store offload for >8 MiB payloads — hook via @nats-io/obj.
  • Dashboard/UI, non-TS SDKs — but the §3 wire layout is frozen v:1 and language-neutral.
  • Separate activity workers — ctx.run steps run inline in the worker process.

#11. Verification strategy (the mechanics test suite you asked for)

Phase 1 ships these as a real vitest suite (unit = pure; integration = spawned nats-server -js). The Appendix-A probe graduates into the integration layer.

  • Primitives (codify Appendix A): journal append CAS fencing; KV lease create/update CAS; msgID dedup; native schedule delivery + atomic cancellation; per-key TTL expiry.
  • Replay: completed steps return recorded values without re-executing fn (instrument fn with an external call-counter); determinism divergence → fails loud.
  • Crash (the robustness core): kill the worker between step-begin and step-end → restart → fn re-runs (at-least-once) but exactly one step-end lands; kill after step-end → replay skips the step (exactly-once).
  • Concurrency: two workers race the same run → exactly one advances each step (zombie fenced at step-begin, no double-fire asserted via the side-effect counter); force AckWait expiry → no double-advance.
  • Suspension: sleep suspends + resumes on schedule; await suspends + resumes on signal; early-signal handled; call suspends parent + resumes on child completion; signal ordering preserved.
  • Lifecycle: duplicate start (same runId) → one run; result() resolves on terminal; status() tracks state.
  • Property/fuzz (stretch): randomized crash injection across a multi-step run; assert journal invariants (monotonic seq, ≤1 step-end per step, terminal-once) and a deterministic final result.

API facts pinned for implementation:

  • js.publish(subj, data, { msgID, expect:{ lastSubjectSequence }, ttl?, schedule? }) → PubAck{ seq, duplicate }.
  • Lose-the-race error: JetStreamApiError, .code === 10071 (StreamWrongLastSequence).
  • KV: kv.create(k, v, markerTTL?) → rev (first-wins); kv.update(k, v, rev) → rev (CAS); kv.get(k,{revision?}) may be replica-served (Bucket.direct) → never gates progress.
