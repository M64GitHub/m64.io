Coming from Sidekiq

Sidekiq runs background jobs fast and reliably — but jobs are fire-and-forget and at-least-once, so a multi-step process means hand-chaining jobs and rolling your own idempotency. A durable workflow makes that sequence a single function that survives a crash mid-flight.

A note on language. Sidekiq is Ruby; @synadia-ai/durable is a TypeScript library today. The wire format is language-neutral (a Ruby SDK could ride the same NATS), but it doesn't exist yet — so “replace” here means running that workflow on a small TypeScript worker alongside your Ruby app, on the NATS you already operate.

#A job → a durable step

A Sidekiq job class becomes a ctx.run step inside a workflow. perform_async becomes client.start; the sidekiq process becomes a worker. Redis is replaced by your NATS.

Sidekiq · Ruby
ruby
class ChargeJob
  include Sidekiq::Job

  def perform(cart_id, amount)
    Stripe::Charge.create(amount: amount, source: cart_id)
  end
end

# enqueue
ChargeJob.perform_async("c_1", 42)

# run the processor:  $ bundle exec sidekiq
@synadia-ai/durable
durable
import { connect } from "@nats-io/transport-node";
import { createClient, createWorker, defineWorkflow } from "@synadia-ai/durable";

const charge = defineWorkflow("charge", async (ctx, input: { cartId: string; amount: number }) =>
  ctx.run("charge-card", () =>
    stripe.charges.create({ amount: input.amount, source: input.cartId }),
  ),
);

const nc = await connect({ servers: "nats://127.0.0.1:4222" });

// enqueue
const client = createClient({ nc });
await client.start(charge, { cartId: "c_1", amount: 42 });

// run the processor
await createWorker({ nc, workflows: [charge] }).start();

#Retries → built-in, with real idempotency

Sidekiq retries are at-least-once, so the docs (rightly) tell you to make jobs idempotent — by hand. ctx.run retries too, and hands you a stable idempotency key (ctx.key()) so a non-idempotent effect is safe across every retry and redelivery.

Sidekiq · Ruby
ruby
class ChargeJob
  include Sidekiq::Job
  sidekiq_options retry: 5

  def perform(order_id, amount)
    # at-least-once: make this idempotent yourself
    Gateway.charge(order_id, amount)
  end
end
@synadia-ai/durable
durable
await ctx.run(
  "charge",
  () => gateway.charge(orderId, amount, { idempotencyKey: ctx.key() }),
  { retry: { kind: "exponential", attempts: 5, baseMs: 2000, factor: 2 } },
);

#Scheduled → ctx.sleep

A durable timer — the wake is a scheduled NATS message, so a parked run survives restarts and ties up no worker while it waits.

Sidekiq · Ruby
ruby
# run in one hour
ReminderJob.perform_in(1.hour, user_id)
# …or at a specific time
ReminderJob.perform_at(deadline, user_id)
@synadia-ai/durable
durable
await ctx.sleep("cool-off", 60 * 60 * 1000);
await ctx.run("reminder", () => sendReminder(userId));

#Chained jobs → one crash-safe workflow

Sidekiq jobs don't compose into a durable sequence — you enqueue the next job at the end of each one, hand-rolling state and idempotency, and a crash between jobs can drop the rest. A durable workflow is one async function; on a crash a fresh worker replays the journal and resumes at the first unfinished step.

Sidekiq · Ruby
ruby
class ReserveJob
  include Sidekiq::Job
  def perform(cart, amount)
    order_id = reserve(cart)
    AuthorizeJob.perform_async(order_id, amount)  # crash here → lost
  end
end
# …and an AuthorizeJob that enqueues CaptureJob, and so on.
@synadia-ai/durable
durable
const checkout = defineWorkflow("checkout", async (ctx, input: Cart) => {
  const orderId = await ctx.run("reserve", () => reserve(input.cart));
  const authId = await ctx.run("authorize", () => authorize(orderId, input.amount));
  return ctx.run("capture", () => capture(authId));
});

// crash anywhere? replay returns completed steps from the journal and
// resumes at the first unfinished one. Exactly-once state.

#Cron → continue-as-new

Instead of sidekiq-cron, a long-running workflow does its work, sleeps a period, and continueAsNews into the next cycle — “run forever” with a journal that stays bounded.

Sidekiq · Ruby
ruby
# config/schedule.yml (sidekiq-cron)
nightly_rollup:
  cron: "0 0 * * *"
  class: "RollupJob"
@synadia-ai/durable
durable
const rollup = defineWorkflow("rollup", async (ctx, s: { day: number }) => {
  await ctx.run("rollup", () => doRollup(s.day));
  await ctx.sleep("one-day", 24 * 60 * 60 * 1000);
  return ctx.continueAsNew({ day: s.day + 1 });
});

#When durable fits — and when Sidekiq is still the tool

  • Reach for durable when the work is a multi-step process that must not lose a step or repeat an effect across a crash — checkout, payments, provisioning, sagas, approvals.
  • Stay on Sidekiq for high-throughput, independent background jobs in a Ruby app, where its speed, simplicity, and ecosystem (web UI, Pro/Enterprise) are exactly what you want. durable is a durable-execution library, not a drop-in job processor.
  • Fan-out. Batch-style fan-out maps to ctx.startCall per child + native Promise.all / ctx.race — durable fan-out is built in. (Per-branch local-step journaling, ctx.startRun, is the deferred bit; wrap parallel local work in one ctx.run(() => Promise.all([...])).)
  • Language. It's TypeScript today; a Ruby SDK over the same wire format is a future, not a shipped, thing.
