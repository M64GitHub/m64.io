Generate a Ghibli-style image

AI image generation is slow, costly, and non-idempotent — the textbook case for durable execution. Wrap the OpenAI call in a ctx.run step and a crash, retry, or redelivery can never make you pay for the same image twice.

This uses the openai Node SDK for the model calls and @synadia-ai/durable for durability. putObject / notifyUser are your own helpers (S3 / R2 / GCS, email / webhook). You'll need an OPENAI_API_KEY and a running NATS server.

#Why durable execution fits image generation

  • It's expensive. Every generation costs money and tens of seconds. Re-running it on a crash or retry is exactly what you must avoid — durable steps run the effect at-least-once but journal the result exactly once, and ctx.key() makes the API call itself idempotent.
  • It's multi-step. Generate → store → notify → (maybe) wait for approval → regenerate. That's a workflow, not a fire-and-forget job — and it must survive a restart anywhere in the middle.
  • It waits on humans. “Approve or try again” can park for hours; a durable ctx.await holds that with no worker tied up.

#The workflow

Two durable steps: generate the image, then store it. The run returns a short URL.

ts
import OpenAI from "openai";
import { defineWorkflow } from "@synadia-ai/durable";

const openai = new OpenAI(); // reads OPENAI_API_KEY from the env

const ghibliPrompt = (subject: string) =>
  `${subject} — in the warm, hand-painted style of a Studio Ghibli film: ` +
  `soft natural light, lush painterly backgrounds, gentle colour, a quiet sense of wonder`;

export const ghibli = defineWorkflow(
  "ghibli-image",
  async (ctx, input: { subject: string; userId: string }) => {
    // 1) Generate — slow, costly, NON-idempotent. Journaled exactly once and
    //    keyed on ctx.key(), so a retry or redelivery never bills you twice.
    const png = await ctx.run(
      "generate",
      async () => {
        const res = await openai.images.generate(
          {
            model: "gpt-image-1",
            prompt: ghibliPrompt(input.subject),
            size: "1024x1024",
            quality: "high",
          },
          { idempotencyKey: ctx.key(), timeout: 120_000 },
        );
        return res.data[0]!.b64_json!; // gpt-image-1 returns base64
      },
      { retry: { kind: "exponential", attempts: 4, baseMs: 1_000, factor: 2 } },
    );

    // 2) Store the bytes durably; the run returns only a short URL.
    const url = await ctx.run("store", () =>
      putObject(`ghibli/${ctx.runId}.png`, Buffer.from(png, "base64")),
    );

    return { url };
  },
);

#Run it

Register the workflow on a worker, then start a run from your app and await the result.

ts
import { connect } from "@nats-io/transport-node";
import { createClient, createWorker } from "@synadia-ai/durable";

const nc = await connect({ servers: "nats://127.0.0.1:4222" });

// A worker runs the code. Image generation can take a while, so give the step
// more room than the 30 s default AckWait (see the note below).
await createWorker({ nc, workflows: [ghibli], ackWaitMs: 120_000 }).start();

// Kick off a run from your API and await the result (which survives restarts).
const client = createClient({ nc });
const handle = await client.start(ghibli, { subject: "a seaside town at dusk", userId: "u_1" });

const { url } = await handle.result(180_000);
console.log("ghibli image:", url);

#What crash-replay buys you here

Say the worker dies after generate but before store. A fresh worker reloads the journal and replays generate from the log — returning the already-made image without calling (or paying) OpenAI again — then runs only store and finishes. The billable call happens once, across the crash:

Contrast a plain job queue, where a retry simply re-runs your task — and a non-idempotent image call is billed a second time. The durable step replays the result instead of re-executing it:

That's exactly-once state, at-least-once effects; the OpenAI Idempotency-Key (set to ctx.key()) covers the rarer case where a step genuinely re-runs in flight (e.g. an AckWait redelivery while the call is still in progress).

#Production notes (the honest edges)

  • Long calls vs. AckWait. A step must finish within the worker's AckWait or the work item is redelivered and the step re-runs (safe, thanks to the idempotency key — just wasteful). The default is 30 s; high-quality generations can exceed that, so set ackWaitMs above your worst-case step time (we used 120 s). There's no working() to extend AckWait mid-step in v0.1.
  • Keep bytes out of the journal at scale. A 1024² PNG is ~1–2 MB of base64. It fits the 8 MiB entry ceiling, but journaling raw bytes bloats the log. For volume, upload inside the generate step and journal only the key:
ts
// Upload INSIDE the generate step and journal only the key — the 1–2 MB of
// base64 never touches the log. (Trade-off: a crash before the upload re-runs
// the whole step, i.e. re-generates.)
const url = await ctx.run("generate-and-store", async () => {
  const res = await openai.images.generate(
    { model: "gpt-image-1", prompt: ghibliPrompt(input.subject), size: "1024x1024" },
    { idempotencyKey: ctx.key() },
  );
  const bytes = Buffer.from(res.data[0]!.b64_json!, "base64");
  return putObject(`ghibli/${ctx.runId}.png`, bytes); // returns a short URL
});
  • Retries. The { retry } policy on ctx.run handles transient OpenAI errors (rate limits, 5xx); pair it with the SDK's own maxRetries if you like. A terminal failure fails the run (catch it to take a fallback path).

#Human-in-the-loop: regenerate until approved

Show the user each take and let them approve or ask for another. The run parks on ctx.await between takes (no worker held), and each attempt's ctx.key() is distinct so every regeneration is billed exactly once. Same-named awaits are matched to signals in arrival order (FIFO), so one signal advances one loop iteration. 

ts
export const ghibliReview = defineWorkflow(
  "ghibli-review",
  async (ctx, input: { subject: string; userId: string }) => {
    for (let attempt = 0; ; attempt++) {
      const url = await ctx.run(`render-${attempt}`, async () => {
        const res = await openai.images.generate(
          { model: "gpt-image-1", prompt: ghibliPrompt(input.subject), size: "1024x1024" },
          { idempotencyKey: ctx.key() }, // unique per attempt → each take billed once
        );
        const bytes = Buffer.from(res.data[0]!.b64_json!, "base64");
        return putObject(`ghibli/${ctx.runId}-${attempt}.png`, bytes);
      });

      await ctx.run(`notify-${attempt}`, () => notifyUser(input.userId, url));

      // Park here — consuming NO worker — until the user approves or asks again.
      const review = await ctx.await<{ approved: boolean }>("review", {
        timeoutMs: 24 * 60 * 60 * 1000,
      });
      if (review.approved) return { url, attempts: attempt + 1 };
    }
  },
);
ts
// From your API, when the user clicks a button on the preview:
await client.signal(runId, "review", { approved: false }); // → render another take
await client.signal(runId, "review", { approved: true }); //  → done, resolve the run

#Variant: turn a photo into Ghibli style

Swap text-to-image for an image edit — same durability, same idempotency.

ts
import OpenAI, { toFile } from "openai";

// "Ghiblify" an uploaded photo (image edit) instead of text-to-image:
const png = await ctx.run("ghiblify", async () => {
  const res = await openai.images.edit(
    {
      model: "gpt-image-1",
      image: await toFile(photoBytes, "photo.png", { type: "image/png" }),
      prompt: "Re-render this photo in the style of a Studio Ghibli film",
    },
    { idempotencyKey: ctx.key() },
  );
  return res.data[0]!.b64_json!;
});
