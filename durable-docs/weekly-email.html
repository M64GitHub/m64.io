Send a weekly email

A recurring job — “email every subscriber once a week” — is one workflow that sends, sleeps a week, and renews itself. No external cron, no scheduler service, no heartbeat process: the durable timer and the run loop are the schedule.

buildDigest, sendEmail, and isSubscribed are your own helpers (your DB + an email provider like Resend / SES / Postmark). Everything else is the real @synadia-ai/durable API.

#Why durable execution fits a weekly job

  • It just survives. A normal cron/timer is lost if the process restarts at the wrong moment. ctx.sleep is a scheduled NATS message — the wake fires after a worker restart, a deploy, or a week-long gap, because nothing in your process is holding the timer.
  • Exactly-once per week. Wrapping the send in ctx.run + ctx.key() means a crash or redelivery around the send never emails the subscriber twice.
  • Runs forever, stays small. ctx.continueAsNew ends each week's run and starts the next with an empty journal — the log never grows without bound, however many years it runs.
  • Costs nothing while idle. A run parked on a week-long sleep holds no worker; you can have millions of parked weekly chains.

#The workflow

Build the digest, send it (exactly once), sleep a week, then renew. That's the whole thing.

ts
import { defineWorkflow } from "@synadia-ai/durable";

const WEEK_MS = 7 * 24 * 60 * 60 * 1000;

export const weeklyDigest = defineWorkflow(
  "weekly-digest",
  async (ctx, input: { to: string; week: number }) => {
    // Build this week's content. Any I/O goes through ctx.run so it's journaled.
    const html = await ctx.run("build", () => buildDigest(input.to, input.week));

    // Send it — keyed on ctx.key() so a crash or redelivery never double-sends.
    await ctx.run("send", () =>
      sendEmail(
        { to: input.to, subject: `Your weekly digest #${input.week}`, html },
        { idempotencyKey: ctx.key() },
      ),
    );

    // Wait a week. The wake is a scheduled NATS message: it survives restarts,
    // and the run holds no worker while it's parked.
    await ctx.sleep("one-week", WEEK_MS);

    // Start next week fresh, keeping the journal bounded. Runs forever.
    return ctx.continueAsNew({ to: input.to, week: input.week + 1 });
  },
);

#Run it

Register the workflow and start one chain per subscriber. Because start is idempotent on the runId, your “subscribe” endpoint can be called any number of times safely. 

ts
import { connect } from "@nats-io/transport-node";
import { createClient, createWorker } from "@synadia-ai/durable";

const nc = await connect({ servers: "nats://127.0.0.1:4222" });

await createWorker({ nc, workflows: [weeklyDigest] }).start();

// One chain per subscriber. The runId is the idempotency key for start, so a
// double-subscribe is a harmless no-op. (runId must match [A-Za-z0-9_-]+.)
const client = createClient({ nc });
await client.start(weeklyDigest, { to: "ada@example.com", week: 1 }, { runId: "weekly-ada" });

#“Run forever”, with a bounded journal

Each week is its own short-lived run; continueAsNew hands off to the next generation with a fresh journal, so no single run's log grows over time. The original handle.result() transparently follows the chain — though a forever-loop never returns until you stop it (next section).

#Never double-send

If the worker crashes after the email is sent but before the journal records it, the resumed run could try to send again. Keying the call on ctx.key() (${runId}:${seq}, stable across re-runs) makes the second attempt a no-op at your email provider — most have an idempotency key or a dedupe window. And if it crashes before sending, replay simply runs the send step for the first time. Either way: one email per week.

#Send on a real schedule (e.g. every Monday 9am)

A flat seven-day sleep drifts relative to wall-clock time. To pin a weekday and hour, compute the delay from ctx.now() — the journaled timestamp — so the math stays deterministic on replay. 

ts
// Want a real cadence (every Monday 09:00 UTC) instead of a flat 7 days?
// Compute the delay from ctx.now() — a pure function of the journaled
// timestamp, so it stays deterministic across replays.
function msUntilNextMonday(now: number): number {
  const d = new Date(now);
  const daysUntilMon = ((1 - d.getUTCDay() + 7) % 7) || 7; // 1 = Monday; never today
  const next = Date.UTC(d.getUTCFullYear(), d.getUTCMonth(), d.getUTCDate() + daysUntilMon, 9);
  return next - now;
}

// …then in the workflow, replace the flat sleep with:
const now = await ctx.now();
await ctx.sleep("until-monday", msUntilNextMonday(now));

#Letting people unsubscribe (and stopping a chain)

There's no client.cancel in v0.1, so a recurring chain isn't stopped from the outside — it ends by choosing not to continue. Check the subscription each cycle (or ctx.await an "unsubscribe" signal) and simply return instead of calling continueAsNew: 

ts
// There's no client.cancel in v0.1 — a forever-loop ends by NOT continuing.
// Check each cycle (or handle an "unsubscribe" signal) and just return:
const subscribed = await ctx.run("check-subscription", () => isSubscribed(input.to));
if (!subscribed) return { stopped: true, lastWeek: input.week };

return ctx.continueAsNew({ to: input.to, week: input.week + 1 });

This is the pattern behind any recurring job — billing runs, nightly rollups, reminders, retention emails. See how state is persisted, the AI-image recipe for the exactly-once story in depth, and the cron walkthrough.
