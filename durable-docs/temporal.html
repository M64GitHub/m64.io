Coming from Temporal

The mental model is the same — workflows, activities, signals, timers, child workflows, continue-as-new. The difference is the backend: no Temporal cluster, no database, just the NATS you already run. Here's how the everyday patterns translate.

These map Temporal's TypeScript SDK to the equivalent @synadia-ai/durable code. The durable snippets use the real API; the Temporal snippets are idiomatic and illustrative. This is a focused — see what doesn't map yet for the honest edges.

#Workflow, worker & client

A workflow is an ordinary async function. You register it on a worker and start runs with a client — no separate workflow source path, task-queue wiring, or running cluster to point at beyond your NATS connection.

Temporal · TypeScript SDK
temporal
import { proxyActivities } from "@temporalio/workflow";
import { Worker } from "@temporalio/worker";
import { Client } from "@temporalio/client";
import type * as activities from "./activities";

const { reserve, authorize, capture } =
  proxyActivities<typeof activities>({ startToCloseTimeout: "1m" });

export async function charge(input: CartInput) {
  const orderId = await reserve(input.cartId);
  const authId = await authorize(orderId, input.amount);
  return { receipt: await capture(authId) };
}

// worker.ts
const worker = await Worker.create({
  workflowsPath: require.resolve("./workflows"),
  activities, taskQueue: "charge",
});
await worker.run();

// client.ts
const client = new Client();
const handle = await client.workflow.start(charge, {
  taskQueue: "charge", workflowId: "c_1", args: [input],
});
console.log(await handle.result());
@synadia-ai/durable
durable
import { connect } from "@nats-io/transport-node";
import {
  createClient, createWorker, defineWorkflow,
} from "@synadia-ai/durable";

const charge = defineWorkflow("charge", async (ctx, input: CartInput) => {
  const orderId = await ctx.run("reserve", () => reserve(input.cartId));
  const authId = await ctx.run("authorize", () => authorize(orderId, input.amount));
  return { receipt: await ctx.run("capture", () => capture(authId)) };
});

const nc = await connect({ servers: "nats://127.0.0.1:4222" });

// worker
const worker = createWorker({ nc, workflows: [charge] });
await worker.start();

// client
const client = createClient({ nc });
const handle = await client.start(charge, input, { runId: "c_1" });
console.log(await handle.result());

#Activities → ctx.run

Anything nondeterministic or side-effecting runs inside ctx.run — the equivalent of a Temporal activity. Both give at-least-once execution; with durable you make non-idempotent effects safe by keying on ctx.key(). Inline retry is per-step.

Temporal · TypeScript SDK
temporal
// activities.ts — a plain function
export async function chargeCard(amount: number) {
  return stripe.charges.create({ amount, currency: "usd" });
}

// in the workflow
const { chargeCard } = proxyActivities<typeof activities>({
  startToCloseTimeout: "30s",
  retry: { maximumAttempts: 5, backoffCoefficient: 2 },
});
const charge = await chargeCard(amount);
@synadia-ai/durable
durable
const charge = await ctx.run(
  "charge-card",
  () =>
    stripe.charges.create(
      { amount, currency: "usd" },
      { idempotencyKey: ctx.key() }, // exactly-once effect
    ),
  { retry: { kind: "exponential", attempts: 5, baseMs: 200, factor: 2 } },
);

Activity heartbeats → automatic. Temporal long activities call heartbeat() to prove liveness; here there is nothing to call — the worker extends the item's ack deadline (ack-progress every ackWaitMs / 3) for as long as the step runs, capped by maxWorkMs (default 10 × AckWait) so a genuinely hung step still fails over to another worker.

#Timers → ctx.sleep

Durable sleep — the wake survives restarts because it's a scheduled NATS message, not an in-process timer.

Temporal · TypeScript SDK
temporal
import { sleep } from "@temporalio/workflow";

await sleep("1 day");
await reminderActivity();
@synadia-ai/durable
durable
await ctx.sleep("cool-off", 24 * 60 * 60 * 1000);
await ctx.run("reminder", () => sendReminder());

#Signals → ctx.await + client.signal

Park a run until an external event arrives — an approval, a webhook, a human decision. With durable, the awaited value is the signal payload; add timeoutMs to race a durable timer and take a fallback path.

Temporal · TypeScript SDK
temporal
import { defineSignal, setHandler, condition } from "@temporalio/workflow";

const approve = defineSignal<[Approval]>("approve");

export async function expense(input: Expense) {
  let approval: Approval | undefined;
  setHandler(approve, (a) => { approval = a; });

  const got = await condition(() => !!approval, "3 days");
  if (!got) return reject("timed out");
  return approval!.ok ? pay(input) : reject(approval!.reason);
}

// elsewhere
await handle.signal(approve, { ok: true });
@synadia-ai/durable
durable
const expense = defineWorkflow("expense", async (ctx, input: Expense) => {
  try {
    const approval = await ctx.await<Approval>("approve", {
      timeoutMs: 3 * 24 * 60 * 60 * 1000,
    });
    return approval.ok
      ? ctx.run("pay", () => pay(input))
      : reject(approval.reason);
  } catch {
    return reject("timed out"); // DurableTimeoutError
  }
});

// elsewhere
await client.signal(runId, "approve", { ok: true });

#Child workflows → ctx.call

Compose workflows. ctx.call starts a child run and resumes the parent with its result; a child failure fails the parent.

Temporal · TypeScript SDK
temporal
import { executeChild } from "@temporalio/workflow";

const shipment = await executeChild(ship, {
  args: [{ orderId }],
  workflowId: `ship-${orderId}`,
});
@synadia-ai/durable
durable
const shipment = await ctx.call(ship, { orderId });
// deterministic child runId: `${parentRunId}_${seq}` (override via opts.runId)

#Parallel children & fan-out → ctx.startCall + Promise.all / ctx.race

Temporal fans out by starting several children (or activities) and awaiting them together. The durable equivalent: ctx.startCall starts a child without blocking and returns a DurableFuture, so native Promise.all / Promise.allSettled await them concurrently, and ctx.race takes the first to finish (the safe alternative to Promise.race, which is unsafe over durable futures).

Temporal · TypeScript SDK
temporal
import { executeChild } from "@temporalio/workflow";

const results = await Promise.all(items.map((item) => executeChild(processItem, { args: [item] })));
@synadia-ai/durable
ts
// Start every child in one pass — none blocks — then await them together.
const futures = items.map((item) => ctx.startCall(processItem, item));
const results = await Promise.all(futures); // suspends until ALL children resolve

// …or race them and take the first to finish (losing branches run to completion):
const winner = await ctx.race(items.map((item) => ctx.startCall(processItem, item)));

ctx.startSleep / ctx.startAwait fan out timers and signals the same way; ctx.call / ctx.sleep / ctx.await are just await ctx.startCall / startSleep / startAwait.

#Continue-as-new → ctx.continueAsNew

Run “forever” with a bounded log: end this generation and start a fresh one with an empty journal. handle.result() transparently follows the chain.

Temporal · TypeScript SDK
temporal
import { continueAsNew, sleep } from "@temporalio/workflow";

export async function subscription(s: Sub) {
  await chargeActivity(s.account);
  await sleep("30 days");
  await continueAsNew<typeof subscription>({
    ...s, cycle: s.cycle + 1,
  });
}
@synadia-ai/durable
durable
const subscription = defineWorkflow("subscription", async (ctx, s: Sub) => {
  await ctx.run("charge", () => charge(s.account));
  await ctx.sleep("billing-period", 30 * 24 * 60 * 60 * 1000);
  return ctx.continueAsNew({ ...s, cycle: s.cycle + 1 });
});

#Cancellation → client.cancel + ctx.onCancel

Stop a run from the outside. client.cancel(runId) delivers an uncatchable hard-stop at the run's next suspend point — a broad try/catch can't swallow it — and the run ends cancelled. Cleanup is opt-in via ctx.onCancel (handlers run LIFO, non-cancellably), and cancellation cascades to in-flight children (Temporal's ParentClosePolicy: REQUEST_CANCEL).

Temporal · TypeScript SDK
temporal
import { CancellationScope, isCancellation } from "@temporalio/workflow";

try {
  await shipOrder(order);
} catch (e) {
  if (isCancellation(e)) {
    await CancellationScope.nonCancellable(() => refund(order));
  }
  throw e;
}

// elsewhere
await handle.cancel();
@synadia-ai/durable
ts
const order = defineWorkflow("order", async (ctx, input: Order) => {
  ctx.onCancel(() => ctx.run("refund", () => refund(input))); // opt-in compensation; runs on cancel
  await ctx.run("charge", () => charge(input));
  await ctx.await("ship-confirmation"); // ← a cancel arriving here hard-stops the run → `cancelled`
});

// elsewhere — an operational kill switch; cascades to in-flight ctx.call children
await client.cancel(runId, { reason: "customer withdrew" });

The key difference: cancellation is uncatchable by design — a main-flow try/finally does not run on cancel, so cleanup goes through ctx.onCancel. Temporal's fine-grained cancellation scopes have no equivalent yet.

#Determinism → ctx.now / random / uuid

Both runtimes require workflow code to be deterministic and re-run it to recover. Reach for the context-provided builtins instead of ambient nondeterminism; durable fails loud if a replay ever diverges from the journal.

Temporal · TypeScript SDK
temporal
import { uuid4 } from "@temporalio/workflow";

const id = uuid4();
const now = Date.now();   // patched to be deterministic
// Math.random() is disallowed
@synadia-ai/durable
durable
const id = await ctx.uuid();
const now = await ctx.now();
const r = await ctx.random();
// bare Date.now() / Math.random() in the body → divergence (caught)

#What doesn't map (yet)

Being honest about the edges — @synadia-ai/durable is a focused, not a drop-in Temporal replacement for every feature:

  • Cancellation scopes. client.cancel + ctx.onCancel (with a cascade to in-flight children) are built in, but Temporal's fine-grained CancellationScopes — and a suspending cleanup handler (a child / sleep inside ctx.onCancel) — have no equivalent yet.
  • Rich queries. No arbitrary defineQuery handlers — observability is handle.status() plus reading the journal (and the live dashboard).
  • Versioning. Version-pinned dispatch: register multiple (name, version) of a workflow (defineWorkflow(name, fn, { version })); each run executes the version it started under, and ctx.continueAsNew(input, { version }) migrates a chain. Temporal's in-code patching (patched() / getVersion() — evolving an in-flight run in place within one version) has no equivalent yet.
  • Search attributes & visibility queries are documented non-goals for now. (Payloads bigger than the NATS message limit offload to the DE_BLOBS object store automatically — the analog of Temporal's payload codec + external storage, built in.)

What you do get is the correctness core — durable steps, timers, signals, child workflows, durable fan-out, cancellation, continue-as-new, crash-replay with exactly-once state — on infrastructure you already operate. See the model, how state is persisted, and the runnable examples/.
