Virtual keys & the vault

The agent never holds the real key. The store never holds a usable one. Revocation is a KV delete — live in milliseconds.

In transparent mode (the default), agents carry real provider keys and the proxy forwards them untouched — fine for a dev box, harder at fleet scale: keys get shared across agents, attribution blurs, and revoking one agent means rotating everyone. Virtual keys fix all three at once, per request, with no mode switch.

#1. The shape of a virtual key

  • Format: sk-syn-v1-<32 bytes base64url> — minted by the admin CLI, printed exactly once. The proxy stores only its SHA-256; a KV compromise leaks no usable keys.
  • Detection is per-request: an sk-syn- prefix in Authorization: Bearer … or x-api-key makes the request virtual; anything else is transparent-mode and forwarded untouched, exactly as the MVP contract. The two coexist with no configuration.
  • For the agent it's just an API key: export OPENAI_API_KEY='sk-syn-v1-…' — no SDK changes, no headers, no awareness.

The store, live — one virtual key exists, and this is everything an attacker with the KV would see:

sh
nats kv ls MODELPROXY_KEYS
text
fd8c803f9db6d141ccd58ddd78c6ac7e1588ea0a5fc27df447f10857ba1c1ecd

#3. The vault

  • Provider keys rest AES-256-GCM-sealed in KV, under a 32-byte master key from --vault-key-file or $MODELPROXY_VAULT_KEY — never argv, and never stored in NATS. The sealed blob plus the bucket is useless without the file.
  • Generate it once and keep it safe — losing it orphans every sealed provider key:
sh
modelproxy admin vault-key generate > vault.key
# store this in a file for --vault-key-file (or MODELPROXY_VAULT_KEY);
# losing it orphans every sealed provider key

PROVIDER_API_KEY="$OPENROUTER_API_KEY" modelproxy admin provider-key set \
    --owner m64 --provider openrouter --vault-key-file vault.key --js-url nats://127.0.0.1:4222
text
sealed provider key stored for m64/openrouter (101 bytes sealed)

Secrets never ride argv: the provider key comes from $PROVIDER_API_KEY or stdin, so it can't leak through process listings or shell history.

  • Wrong or absent vault key ⇒ virtual requests answer 503 policy_unavailable (fail-closed, detected loudly at boot); transparent traffic is untouched.

#4. The identity binding

A key's {owner, agent} are authoritative — the URL path must match, or the request is denied. This stops key borrowing across agents cold. Captured live: the key minted for m64/claude-code, presented on a m64/codex path: 

text
HTTP/1.1 403 Forbidden
{"error":{"message":"denied by synadia-model-proxy policy (key.identity)","rule":"key.identity","type":"key_identity_mismatch"}}
  • session stays free — it names an instance, not a principal; one key serves an agent's many sessions with per-session attribution intact.
  • An optional per-key --providers allowlist restricts which upstreams the key can reach at all (403, rule key.providers) — including llm.http traffic.

#5. Lifecycle: mint, list, revoke

sh
modelproxy admin key create --owner m64 --agent claude-code --providers openrouter \
    --js-url nats://127.0.0.1:4222
text
id:  vk_0Bmb0vVCD4Ah1kvoXguqid
key: sk-syn-v1-qe87qLl2VGQeZnnoFDbUJ3HPz9GF5k93VChwk9qZMmM
this key is shown ONCE — only its hash is stored
sh
modelproxy admin key list --owner m64
text
ID                         OWNER            AGENT            PROVIDERS  DISABLED EXPIRES
vk_0Bmb0vVCD4Ah1kvoXguqid  m64              claude-code      openrouter false    -
  • key disable / enable / delete --id vk_…, plus --expires 2160h at mint time for self-expiring keys.
  • Revocation propagates through the KV watch in milliseconds — no restart, no cache to flush. The disabled key's very next call answers 401 invalid_virtual_key. (Proven live in the walkthrough; the key above was revoked right after that capture.)

#6. What records show, and how it fails

  • Records from virtual-key calls carry "auth": "virtual" and the opaque "vk_id" — never key material, in either direction. mutated stays about the body (usage injection); auth replacement is not a mutation of the recorded conversation.
  • Fail-closed by nature: without key material the proxy cannot forward a virtual request — there is nothing to authenticate with upstream. Key mirror never loaded (fresh boot, KV down) ⇒ 503 policy_unavailable; unknown, disabled, or expired key ⇒ 401 invalid_virtual_key.
  • Transparent-mode traffic is never affected by any of this — the failure domains don't overlap.
